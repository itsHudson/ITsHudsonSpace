# 1.4 SQL Command Categories

---

## 1.4.1 Introduction to SQL Command Categories

---

### 1.4.1.1 Why SQL Commands are Categorized

SQL contains many commands.

To make SQL easier to learn and organize:

- SQL commands are grouped into categories

Each category focuses on a different responsibility.

Examples:

| Category | Main Purpose |
|---|---|
| DDL | Define database structure |
| DML | Manipulate data |
| DQL | Query data |
| DCL | Control permissions |
| TCL | Manage transactions |

---

### 1.4.1.2 Real-World Analogy

Think of a database system like:

- A large office building

Different SQL categories act like different departments.

| Department | SQL Category | Responsibility |
|---|---|---|
| Construction team | DDL | Build structures |
| Office workers | DML | Work with data |
| Information desk | DQL | Retrieve information |
| Security department | DCL | Control access |
| Recovery department | TCL | Handle rollback and recovery |

---

### 1.4.1.3 Why Understanding SQL Categories is Important

Understanding command categories helps beginners:

| Benefit | Explanation |
|---|---|
| Organize SQL knowledge | Easier learning |
| Understand database workflows | Better system understanding |
| Improve troubleshooting | Easier debugging |
| Learn permissions and transactions | Better enterprise readiness |

---

### 1.4.1.4 Overview of SQL Command Categories

| Category | Full Name | Purpose |
|---|---|---|
| DDL | Data Definition Language | Define structures |
| DML | Data Manipulation Language | Manage data |
| DQL | Data Query Language | Retrieve data |
| DCL | Data Control Language | Control permissions |
| TCL | Transaction Control Language | Manage transactions |

---

## 1.4.2 DDL (Data Definition Language)

---

### 1.4.2.1 What is DDL

DDL stands for:

- Data Definition Language

DDL commands are used to:

- Create database structures
- Modify database structures
- Remove database structures

DDL mainly works with:

| Database Object | Example |
|---|---|
| Tables | employees |
| Databases | company_db |
| Indexes | idx_customer |
| Views | employee_view |

---

### 1.4.2.2 Real-World Analogy

Think of DDL like:

- Construction work in a building

Examples:

| Construction Activity | SQL Equivalent |
|---|---|
| Build new room | CREATE TABLE |
| Expand room | ALTER TABLE |
| Destroy room | DROP TABLE |
| Rename room | RENAME |

---

## 1.4.2.3 CREATE Command

---

### 1.4.2.3.1 Purpose of CREATE

The CREATE command is used to:

- Create new database objects

Examples:

| Object | Example |
|---|---|
| Database | CREATE DATABASE |
| Table | CREATE TABLE |
| View | CREATE VIEW |

---

### 1.4.2.3.2 CREATE TABLE Syntax

```sql
CREATE TABLE students (
    student_id INT,
    student_name VARCHAR(100),
    age INT
);
```

---

### 1.4.2.3.3 Step-by-Step Explanation

| Step | Explanation |
|---|---|
| CREATE TABLE | Start table creation |
| students | Table name |
| student_id INT | Integer column |
| student_name VARCHAR(100) | Text column |
| age INT | Age column |

---

### 1.4.2.3.4 Result After Creation

#### students Table

| student_id | student_name | age |
|---|---|---|
| Empty | Empty | Empty |

The structure exists even without data.

---

## 1.4.2.4 ALTER Command

---

### 1.4.2.4.1 Purpose of ALTER

The ALTER command is used to:

- Modify existing database structures

---

### 1.4.2.4.2 Common ALTER Operations

| Operation | Example |
|---|---|
| Add column | ADD COLUMN |
| Remove column | DROP COLUMN |
| Modify datatype | MODIFY |
| Rename column | RENAME COLUMN |

---

### 1.4.2.4.3 Example Add Column

```sql
ALTER TABLE students
ADD email VARCHAR(100);
```

---

### 1.4.2.4.4 Before and After Example

#### Before ALTER

| student_id | student_name | age |
|---|---|---|
| 1 | Alice | 20 |

#### After ALTER

| student_id | student_name | age | email |
|---|---|---|---|
| 1 | Alice | 20 | NULL |

---

## 1.4.2.5 DROP Command

---

### 1.4.2.5.1 Purpose of DROP

The DROP command permanently removes:

- Tables
- Databases
- Views
- Other objects

---

### 1.4.2.5.2 Example DROP TABLE

```sql
DROP TABLE students;
```

---

### 1.4.2.5.3 Important Warning

DROP removes:

- Structure
- Data
- Relationships

Everything is permanently deleted.

---

### 1.4.2.5.4 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Confusing DROP and DELETE | DROP removes structure, DELETE removes rows |
| Running DROP accidentally | Can destroy production systems |
| No backup before DROP | Dangerous in real systems |

---

## 1.4.2.6 TRUNCATE Command

---

### 1.4.2.6.1 Purpose of TRUNCATE

The TRUNCATE command:

- Removes all rows from a table

However:

- Table structure remains

---

### 1.4.2.6.2 Example TRUNCATE

```sql
TRUNCATE TABLE students;
```

---

### 1.4.2.6.3 Difference Between DELETE and TRUNCATE

| Feature | DELETE | TRUNCATE |
|---|---|---|
| Removes rows | Yes | Yes |
| Removes structure | No | No |
| WHERE support | Yes | Usually No |
| Faster | Slower | Faster |
| Transaction logging | More | Less |

---

### 1.4.2.6.4 Real-World Analogy

| Action | Meaning |
|---|---|
| DELETE | Remove selected files |
| TRUNCATE | Empty entire cabinet instantly |

---

---

### 1.4.3.1 What is DML

- Data Manipulation Language

- Actual data inside tables

DML focuses on:

| Action | Purpose |
|---|---|
| Insert | Add data |
| Update | Modify data |
| Delete | Remove data |
| Merge | Combine operations |

---

### 1.4.3.2 Real-World Analogy

Think of DML like:

- Employees working with files inside office cabinets

Examples:

| Activity | SQL Command |
|---|---|
| Add new employee file | INSERT |
| Update phone number | UPDATE |
| Remove outdated record | DELETE |

---

## 1.4.3.3 INSERT Command

---

### 1.4.3.3.1 Purpose of INSERT

The INSERT command adds:

- New rows into a table

### 1.4.3.3.2 Example INSERT

INSERT INTO students (
    student_id,

    age

    'Alice',

### 1.4.3.3.3 Result After INSERT

| student_id | student_name | age |
|---|---|---|
| 1 | Alice | 20 |

| Mistake | Explanation |

## 1.4.3.4 UPDATE Command

---

### 1.4.3.4.1 Purpose of UPDATE

The UPDATE command modifies:

- Existing rows

---

### 1.4.3.4.2 Example UPDATE

```sql
UPDATE students
SET age = 21
WHERE student_id = 1;
```

---

### 1.4.3.4.3 Before and After Example

#### Before UPDATE

| student_id | student_name | age |
|---|---|---|
| 1 | Alice | 20 |

#### After UPDATE

| student_id | student_name | age |
|---|---|---|
| 1 | Alice | 21 |

---

### 1.4.3.4.4 Critical Beginner Mistake

#### Dangerous Query

```sql
UPDATE students
SET age = 21;
```

Without WHERE:

- Every row gets updated

This is one of the most dangerous beginner mistakes.

---

## 1.4.3.5 DELETE Command

---

### 1.4.3.5.1 Purpose of DELETE

The DELETE command removes:

- Rows from a table

---

### 1.4.3.5.2 Example DELETE

```sql
DELETE FROM students
WHERE student_id = 1;
```

---

### 1.4.3.5.3 Important Difference

| Command | Removes |
|---|---|
| DELETE | Rows only |
| DROP | Entire table |
| TRUNCATE | All rows quickly |

---

### 1.4.3.5.4 Dangerous Beginner Mistake

#### Dangerous Query

```sql
DELETE FROM students;
```

Without WHERE:

- All rows are deleted

## 1.4.3.6 MERGE Command

---

### 1.4.3.6.1 Purpose of MERGE

The MERGE command combines:

- INSERT
- UPDATE
- DELETE

into one operation.

---

### 1.4.3.6.2 Why MERGE is Useful

MERGE is commonly used for:

- Data synchronization
- ETL processes
- Data warehousing

---

### 1.4.3.6.3 Conceptual MERGE Flow

| Condition | Action |
|---|---|
| Record exists | UPDATE |
| Record missing | INSERT |

---

### 1.4.3.6.4 Simplified Example

```sql
MERGE INTO target_table
USING source_table
ON target_table.id = source_table.id
WHEN MATCHED THEN
    UPDATE SET target_table.name = source_table.name
WHEN NOT MATCHED THEN
    INSERT (id, name)
    VALUES (source_table.id, source_table.name);
```

## 1.4.5 DCL (Data Control Language)

---

### 1.4.5.1 What is DCL

DCL stands for:

- Data Control Language

DCL controls:

- User permissions
- Database access
- Security rules

---

### 1.4.5.2 Why DCL is Important

In enterprise systems:

- Not every user should access all data

Examples:

| User | Allowed Access |
|---|---|
| HR Staff | Employee records |
| Finance Staff | Billing data |
| Customers | Own accounts only |

---

## 1.4.5.3 GRANT Command

---

### 1.4.5.3.1 Purpose of GRANT

The GRANT command gives:

- Permissions to users

---

### 1.4.5.3.2 Example GRANT

```sql
GRANT SELECT
ON students
TO analyst_user;
```

---

### 1.4.5.3.3 Meaning

This allows:

- analyst_user
- to read data from students

---

## 1.4.5.4 REVOKE Command

---

### 1.4.5.4.1 Purpose of REVOKE

The REVOKE command removes:

- User permissions

---

### 1.4.5.4.2 Example REVOKE

```sql
REVOKE SELECT
ON students
FROM analyst_user;
```

---

### 1.4.5.4.3 Real-World Analogy

| Security Activity | SQL Command |
|---|---|

| Remove access card | REVOKE |

## 1.4.6 TCL (Transaction Control Language)

---

### 1.4.6.1 What is TCL

TCL stands for:

- Transaction Control Language

TCL manages:

- Transactions

---

### 1.4.6.2 What is a Transaction

A transaction is:

- A group of SQL operations treated as one unit

Example banking transfer:

| Step | Action |
|---|---|
| Step 1 | Deduct money |
| Step 2 | Add money |
| Step 3 | Confirm transfer |

If one step fails:

- Entire transaction should fail

---

### 1.4.6.3 Why Transactions are Important

Transactions help ensure:

| Feature | Purpose |
|---|---|
| Consistency | Accurate data |
| Reliability | Safe operations |
| Recovery | Rollback failures |

---

## 1.4.6.5 ROLLBACK Command

---

### 1.4.6.5.1 Purpose of ROLLBACK

The ROLLBACK command cancels:

- Uncommitted changes

---

### 1.4.6.5.2 Example ROLLBACK

```sql
ROLLBACK;
```

---

### 1.4.6.5.3 Real-World Analogy

Think of ROLLBACK like:

- Cancelling an online checkout before payment confirmation

---

## 1.4.6.6 SAVEPOINT Command

---

### 1.4.6.6.1 Purpose of SAVEPOINT

A SAVEPOINT creates:

- A checkpoint inside a transaction

---

### 1.4.6.6.2 Why SAVEPOINT is Useful

SAVEPOINT allows:

- Partial rollback

instead of:

- Rolling back entire transaction

---

### 1.4.6.6.3 Example SAVEPOINT

```sql
SAVEPOINT before_update;
```

---

### 1.4.6.6.4 Example Rollback to SAVEPOINT

```sql
ROLLBACK TO before_update;
```

---

### 1.4.6.6.5 Real-World Analogy

Think of SAVEPOINT like:

- A game checkpoint

You can return to:

- The checkpoint
- instead of restarting the entire game

## 1.4.7 SQL Command Workflow Example

---

### 1.4.7.1 Complete Real-World Workflow

#### Step 1 — Create Table (DDL)

```sql
CREATE TABLE employees (
    employee_id INT,
    employee_name VARCHAR(100)
);
```

---

#### Step 2 — Insert Data (DML)

```sql
INSERT INTO employees
VALUES (1, 'Alice');
```

---

#### Step 3 — Retrieve Data (DQL)

```sql
SELECT *
FROM employees;
```

---

#### Step 4 — Grant Access (DCL)

```sql
GRANT SELECT
ON employees
TO analyst_user;
```

---

#### Step 5 — Save Transaction (TCL)

```sql
COMMIT;
```

## 1.4.8 Beginner Mistakes Across SQL Categories

---

### 1.4.8.1 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Forgetting WHERE in UPDATE | Updates all rows |
| Forgetting WHERE in DELETE | Deletes all rows |
| Confusing DROP and DELETE | DROP removes structure |
| Running COMMIT too early | Makes mistakes permanent |
| Ignoring permissions | Security risks |
| Using TRUNCATE carelessly | Removes all data instantly |
