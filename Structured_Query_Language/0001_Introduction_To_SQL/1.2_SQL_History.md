# 1.2 SQL History

Understanding SQL history helps beginners understand:

- why SQL was created
- how databases evolved
- why SQL became a global standard
- why different SQL databases behave differently
- how SQL adapted to cloud computing and big data

Many beginners learn SQL syntax but never understand where SQL came from.

This section explains the evolution of SQL from early database research to modern cloud database systems.

---

# 1.2.1 Origins of SQL

---

## Before SQL Existed

Before databases became common, companies stored records manually.

Examples:

- paper files
- filing cabinets
- spreadsheets
- handwritten ledgers

Problems:

- slow searching
- duplicate records
- data inconsistency
- difficult reporting
- difficult scaling

---

## Early Computer Data Storage

When computers became more common, early systems stored data using:

- flat files
- hierarchical databases
- network databases

These systems were difficult to manage because:

- data relationships were complex
- querying data required programming knowledge
- systems lacked standardization

---

## Edgar F. Codd and the Relational Model

In 1970, IBM researcher:

**Edgar F. Codd**

introduced the:

**Relational Database Model**

---

## What Was Revolutionary?

Instead of storing data in complicated hierarchical structures:

Data could now be stored in:

- rows
- columns
- tables

This became the foundation of modern relational databases.

---

## Real Life Analogy

Old systems:

Like manually navigating complicated folders inside folders inside folders.

Relational model:

Like using organized spreadsheets connected together.

---

## Example Table

| customer_id | customer_name | country |
|-------------|----------------|----------|
| 1 | John | Malaysia |
| 2 | Sarah | Singapore |

This table structure became much easier to understand and query.

---

## Why This Was Important

The relational model solved many problems:

- easier querying
- standardized structure
- reduced duplication
- better scalability
- improved consistency

---

## Key Takeaway

The relational database model became the foundation of SQL databases.

---

# 1.2.2 IBM System R

---

## What is IBM System R?

IBM System R was one of the first relational database research projects.

Created by:

- IBM researchers
- during the 1970s

Purpose:

To test whether relational databases could work in real systems.

---

## Goals of IBM System R

IBM wanted to build a system that could:

- store relational data
- support querying
- improve data management
- reduce programming complexity

---

## Why System R Was Important

System R introduced many concepts still used today:

- SQL querying
- query optimization
- indexing
- transactions
- relational database engines

---

## Beginner Analogy

Think of System R as:

The prototype that proved relational databases actually worked.

Like an early experimental version of modern SQL databases.

---

## Step-by-Step Example

Before System R:

Developers needed complicated low-level programming to retrieve data.

After System R:

Users could write simpler SQL queries.

Example:

```sql
SELECT name
FROM employees;
```

Much easier than low-level data navigation.

---

## Technologies Introduced

| Feature | Importance |
|----------|-------------|
| SQL language | Easier querying |
| Query optimizer | Faster performance |
| Transactions | Reliable updates |
| Indexing | Faster searching |

---

## Key Takeaway

IBM System R proved relational databases were practical and powerful.

---

# 1.2.3 SEQUEL Language

---

## What Was SEQUEL?

Early SQL was originally called:

**SEQUEL**

Meaning:

**Structured English Query Language**

---

## Why SEQUEL Was Created

Goal:

Allow non-programmers to retrieve data using human-readable commands.

Instead of complicated programming syntax.

---

## Example

Instead of low-level programming:

Users could write:

```sql
SELECT name
FROM students;
```

This was revolutionary at the time.

---

## Why the Name Changed

The name SEQUEL was later shortened to:

**SQL**

because of trademark issues.

However:

Many people still pronounce SQL as:

"sequel"

---

## Why SEQUEL Was Important

SEQUEL made databases:

- easier to use
- easier to learn
- more accessible to businesses

---

## Beginner Notes

SQL syntax looks close to English intentionally.

Examples:

- SELECT
- FROM
- WHERE
- ORDER BY

This makes SQL beginner-friendly compared to many programming languages.

---

## Key Takeaway

SEQUEL became the foundation of modern SQL syntax.

---

# 1.2.4 ANSI SQL Standards

---

## What is ANSI?

ANSI stands for:

**American National Standards Institute**

---

## Why SQL Needed Standards

Early database systems behaved differently.

Problem:

A query working in one database might fail in another.

Example:

MySQL syntax might differ from Oracle syntax.

---

## ANSI SQL Goal

ANSI created standard SQL rules so databases could become more consistent.

---

## Benefits of Standards

| Benefit | Explanation |
|----------|-------------|
| Portability | Easier to move systems |
| Consistency | Similar syntax |
| Easier learning | Standardized commands |
| Vendor compatibility | Better interoperability |

---

## Standard SQL Example

```sql
SELECT name
FROM employees
WHERE salary > 5000;
```

Most databases support this because of ANSI standards.

---

## Beginner Notes

Even today:

Different databases still have vendor-specific features.

Examples:

- PostgreSQL features
- Oracle-specific syntax
- SQL Server T-SQL

But ANSI SQL provides the common foundation.

---

## Key Takeaway

ANSI SQL helped standardize SQL across database vendors.

---

# 1.2.5 ISO SQL Standards

---

## What is ISO?

ISO stands for:

**International Organization for Standardization**

---

## ANSI vs ISO

ANSI:

American standards organization

ISO:

International standards organization

---

## Why ISO Standards Matter

SQL became global.

International standardization was needed so:

- companies worldwide could use SQL consistently
- software vendors could follow common rules

---

## ISO SQL Responsibilities

ISO defines:

- SQL syntax standards
- transaction behavior
- data type standards
- query behavior

---

## Example

Common SQL features standardized by ISO:

- SELECT
- INSERT
- UPDATE
- DELETE

---

## Real World Importance

Without standards:

Every database would use completely different query languages.

Developers would struggle switching systems.

---

## Key Takeaway

ISO helped SQL become an international database standard.

---

# 1.2.6 SQL Evolution

---

## SQL Did Not Stay Static

SQL evolved over decades.

New features were added as technology improved.

---

## Early SQL

Focused mainly on:

- simple querying
- relational tables
- basic transactions

---

## Later Improvements

SQL gradually added:

- joins
- subqueries
- views
- triggers
- stored procedures
- window functions
- JSON support

---

## Evolution Timeline

| Period | Major Improvements |
|---------|---------------------|
| 1970s | Relational model |
| 1980s | Commercial SQL databases |
| 1990s | Enterprise databases |
| 2000s | Web application scaling |
| 2010s | Cloud databases |
| 2020s | Distributed SQL & analytics |

---

## Example of Modern SQL

Modern SQL databases now support:

```sql
SELECT customer_id,
       SUM(amount) OVER(PARTITION BY customer_id)
FROM orders;
```

Older SQL systems could not easily perform advanced analytics like this.

---

## Beginner Notes

Modern SQL is much more powerful than early SQL.

Today SQL supports:

- analytics
- JSON
- machine learning integration
- cloud warehousing
- distributed systems

---

## Key Takeaway

SQL evolved continuously to support modern software systems.

---

SQL evolved from simple relational systems into modern cloud-scale data platforms.
