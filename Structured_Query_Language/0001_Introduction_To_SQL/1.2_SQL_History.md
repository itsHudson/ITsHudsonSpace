# 1.2 SQL History

---

## 1.2.1 Origins of SQL

---

### 1.2.1.1 Introduction to SQL Origins

To understand SQL properly, beginners should understand:

- Why SQL was created
- What problems existed before SQL
- How databases evolved over time

SQL did not appear suddenly.

It was developed as part of the evolution of:

- Data storage systems
- Database management systems
- Business information systems
- Computer science research

---

### 1.2.1.2 Data Management Before SQL

Before relational databases existed, many systems used:

- File-based storage systems

Examples:

| Old System | Description |
|---|---|
| Flat files | Data stored in plain text files |
| Hierarchical databases | Tree-like data structures |
| Network databases | Complex linked record systems |

These older systems had many problems.

---

### 1.2.1.3 Problems with Early Database Systems

#### A. Data Duplication

The same data was often repeated many times.

Example:

| File | Duplicate Data |
|---|---|
| Customer file | Customer name |
| Order file | Same customer name again |
| Billing file | Same customer name again |

This caused:

- Wasted storage
- Inconsistent data
- Difficult updates

---

#### B. Difficult Data Access

Finding information was difficult because:

- Developers had to write complex low-level programs
- Data retrieval depended heavily on file structure

Example:

If a company wanted:

- "All customers who ordered this month"

Developers might need:

- Hundreds of lines of procedural code

---

#### C. Poor Flexibility

Changing the database structure was difficult.

Even small changes could:

- Break applications
- Require rewriting programs

---

#### D. Weak Relationships

Older systems struggled to manage:

- Relationships between data

Example:

| Relationship | Example |
|---|---|
| Customer → Orders | One customer has many orders |
| Student → Courses | One student joins many courses |

---

### 1.2.1.4 The Relational Database Revolution

A major breakthrough happened in 1970.

A computer scientist named:

- Edgar F. Codd

published a revolutionary paper called:

- "A Relational Model of Data for Large Shared Data Banks"

This paper introduced:

- The relational model

---

### 1.2.1.5 What is the Relational Model

The relational model introduced the idea that:

- Data should be stored in tables

Each table contains:

| Component | Meaning |
|---|---|
| Rows | Individual records |
| Columns | Data attributes |
| Relationships | Connections between tables |

Example:

#### Customers Table

| customer_id | customer_name |
|---|---|
| 1 | Alice |
| 2 | Bob |

#### Orders Table

| order_id | customer_id |
|---|---|
| 101 | 1 |
| 102 | 2 |

The customer_id connects both tables.

---

### 1.2.1.6 Why the Relational Model Was Important

The relational model solved many problems.

Benefits

| Benefit | Explanation |
|---|---|
| Reduced duplication | Data stored more efficiently |
| Better consistency | Easier updates |
| Easier querying | Simpler data retrieval |
| Better relationships | Connected tables |
| Improved scalability | Better for large systems |

---

### 1.2.1.7 Real-World Analogy

Think of relational databases like:

- Organized Excel spreadsheets with relationships

Instead of storing everything in one giant file:

- Data is separated into logical tables

This makes systems:

- Cleaner
- Faster
- Easier to maintain

---

### 1.2.1.8 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Thinking SQL created databases | Databases existed before SQL |
| Confusing SQL with relational model | SQL is the language, relational model is the design concept |
| Ignoring Edgar Codd | He is considered the father of relational databases |

---

## 1.2.2 IBM System R

---

### 1.2.2.1 Introduction to IBM System R

IBM System R was one of the first major projects to implement:

- Relational database concepts

It became one of the most important projects in database history.

---

### 1.2.2.2 What Was IBM System R

IBM System R was:

- A research database project created by IBM in the 1970s

Purpose:

- To prove that relational databases could work in real-world systems

---

### 1.2.2.3 Main Goals of System R

IBM wanted to show that relational databases could provide:

| Goal | Purpose |
|---|---|
| Efficient querying | Fast data retrieval |
| Scalability | Handle large datasets |
| Multi-user support | Many users simultaneously |
| Data integrity | Reliable information |
| Simpler development | Easier application building |

---

### 1.2.2.4 Major Contributions of System R

System R introduced many concepts still used today.

Important Contributions

| Contribution | Description |
|---|---|
| SQL language | Early SQL implementation |
| Query optimizer | Improved query performance |
| Transactions | Safe data operations |
| Indexing techniques | Faster searching |
| Relational storage | Table-based storage |

---

### 1.2.2.5 Query Optimizer Innovation

One of the most important innovations was:

- The query optimizer

The optimizer decides:

- The best way to execute queries

Example:

```sql
SELECT *
FROM orders
WHERE customer_id = 100;
```

The optimizer determines:

- Whether to use indexes
- Which tables to scan first
- How to reduce processing time

---

### 1.2.2.6 Why System R Was Important

System R proved that:

- Relational databases were practical
- SQL could become a standard language
- Complex queries could be optimized efficiently

Without System R:

- Modern SQL databases might not exist today

---

### 1.2.2.7 Influence on Modern Databases

Many modern database systems were influenced by System R.

Examples:

| Modern Database | Influence |
|---|---|
| DB2 | Direct IBM evolution |
| Oracle | Inspired by relational concepts |
| PostgreSQL | Uses similar relational foundations |
| SQL Server | Relational architecture |

---

## 1.2.3 SEQUEL Language

---

### 1.2.3.1 What Was SEQUEL

Before SQL was called SQL, it was originally named:

- SEQUEL

SEQUEL stands for:

- Structured English Query Language

---

### 1.2.3.2 Why SEQUEL Was Created

The goal was to create a language that:

- Was readable like English
- Allowed non-programmers to query databases
- Simplified database interaction

---

### 1.2.3.3 Example of Human-Readable Design

Example SQL query:

```sql
SELECT name
FROM employees
WHERE department = 'Sales';
```

This resembles natural English:

- "Select name from employees where department is Sales"

This readability became one of SQL's biggest strengths.

---

### 1.2.3.4 Why the Name Changed from SEQUEL to SQL

The name SEQUEL had trademark issues.

IBM shortened the name to:

- SQL

However, many people still pronounce SQL as:

- "Sequel"

---

### 1.2.3.5 Core Philosophy of SQL

SQL follows a declarative philosophy.

This means:

- You describe WHAT you want
- The database decides HOW to retrieve it

---

### 1.2.3.6 Declarative vs Procedural

| Type | Description |
|---|---|
| Declarative | Describe desired result |
| Procedural | Describe step-by-step process |

---

### 1.2.3.7 Example Comparison

#### SQL Declarative Style

```sql
SELECT *
FROM employees
WHERE salary > 5000;
```

#### Procedural Style

A procedural language would require:

- Open file
- Read records
- Compare salary
- Store matching rows
- Return results

SQL simplified this process dramatically.

---

## 1.2.4 ANSI SQL Standards

---

### 1.2.4.1 What is ANSI

ANSI stands for:

- American National Standards Institute

ANSI helps create:

- Technology standards
- Industry standards
- Compatibility standards

---

### 1.2.4.2 Why SQL Standards Were Needed

As SQL became popular:

- Different vendors created different SQL versions

This caused compatibility problems.

Example:

| Vendor | Database |
|---|---|
| IBM | DB2 |
| Oracle | Oracle Database |
| Microsoft | SQL Server |

Without standards:

- Queries might work in one database but fail in another

---

### 1.2.4.3 ANSI SQL Standardization

ANSI created official SQL standards to ensure:

| Goal | Purpose |
|---|---|
| Consistency | Similar syntax |
| Portability | Easier migration |
| Compatibility | Cross-platform support |
| Reliability | Predictable behavior |

---

### 1.2.4.4 First ANSI SQL Standard

The first major ANSI SQL standard was released in:

- 1986

This was a major milestone in database history.

---

### 1.2.4.5 Common ANSI SQL Features

Examples of standardized features:

| Feature | Example |
|---|---|
| SELECT | Query data |
| INSERT | Add rows |
| UPDATE | Modify rows |
| DELETE | Remove rows |
| JOIN | Combine tables |

---

### 1.2.4.6 Example ANSI SQL Query

```sql
SELECT employee_name
FROM employees
WHERE department_id = 10;
```

Most SQL databases support this syntax.

---

### 1.2.4.7 Why ANSI SQL Matters Today

Benefits:

| Benefit | Explanation |
|---|---|
| Easier learning | Similar syntax across databases |
| Easier migration | Move applications between systems |
| Better developer productivity | Shared SQL knowledge |
| Industry stability | Common standards |

---

### 1.2.4.8 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Assuming all SQL is identical | Vendors add custom features |
| Ignoring vendor-specific syntax | PostgreSQL, Oracle, MySQL differ |
| Thinking ANSI SQL covers everything | Vendors extend standards heavily |
