# 1.10 SQL Standards & Compliance

---

## 1.10.1 Introduction to SQL Standards & Compliance

---

### 1.10.1.1 What are SQL Standards

SQL standards are:

- Official rules
- Specifications
- Guidelines

that define how SQL should behave.

These standards help ensure:

| Purpose | Explanation |
|---|---|
| Consistency | Similar SQL behavior |
| Compatibility | Queries work across systems |
| Portability | Easier migration |
| Stability | Predictable database behavior |

---

### 1.10.1.2 Why SQL Standards are Important

Without standards:

- Every database vendor would create completely different SQL syntax

This would create major problems:

| Problem | Explanation |
|---|---|
| Difficult learning | Need to relearn SQL |
| Poor portability | Hard migration |
| Vendor lock-in | Difficult system switching |
| Application incompatibility | Queries fail across systems |

---

### 1.10.1.3 Real-World Analogy

Think of SQL standards like:

- International traffic rules

Although countries may differ slightly:

- Basic traffic rules remain similar

This helps drivers:

- Adapt more easily

Similarly:

- SQL standards help developers work across databases

---

### 1.10.1.4 What is SQL Compliance

SQL compliance means:

- How closely a database follows official SQL standards

Some databases:

- Follow standards very closely

Others:

- Add many proprietary features

---

### 1.10.1.5 Main Topics in SQL Standards

| Topic | Purpose |
|---|---|
| ANSI SQL | American SQL standards |
| ISO SQL | International SQL standards |
| Vendor deviations | Database-specific differences |
| SQL portability | Move SQL across systems |
| Cross-platform compatibility | Support multiple databases |

---

## 1.10.2 ANSI SQL

---

### 1.10.2.1 What is ANSI

ANSI stands for:

- American National Standards Institute

ANSI develops:

- Technology standards
- Industry standards
- Software standards

---

### 1.10.2.2 ANSI SQL Overview

ANSI SQL refers to:

- Standardized SQL specifications created by ANSI

The goal is to ensure:

- SQL consistency across databases

---

### 1.10.2.3 Why ANSI SQL Was Created

Before standardization:

- Every database vendor implemented SQL differently

This caused:

| Problem | Explanation |
|---|---|
| Query incompatibility | Queries failed across systems |
| Learning difficulties | Different syntax everywhere |
| Poor migration support | Hard database switching |

---

### 1.10.2.4 First ANSI SQL Standard

The first ANSI SQL standard was released in:

- 1986

This became a major milestone in database history.

---

### 1.10.2.5 Main Goals of ANSI SQL

| Goal | Explanation |
|---|---|
| Standard syntax | Similar query structure |
| Database portability | Easier migration |
| Cross-platform support | Shared SQL behavior |
| Vendor consistency | Reduced differences |

---

### 1.10.2.6 Example ANSI SQL Query

#### Syntax

```sql
SELECT column1
FROM table_name
WHERE condition;
```

#### Example

```sql
SELECT student_name
FROM students
WHERE age > 20;
```

This syntax works in:

| Database | Supported |
|---|---|
| MySQL | Yes |
| PostgreSQL | Yes |
| SQL Server | Yes |
| Oracle | Yes |

---

### 1.10.2.7 Common ANSI SQL Features

| Feature | Example |
|---|---|
| SELECT | Data retrieval |
| INSERT | Add rows |
| UPDATE | Modify rows |
| DELETE | Remove rows |
| JOIN | Combine tables |

---

### 1.10.2.8 Advantages of ANSI SQL

| Advantage | Explanation |
|---|---|
| Easier learning | Common syntax |
| Better portability | Cross-database migration |
| Reduced vendor lock-in | Flexible architecture |
| Better developer productivity | Shared SQL knowledge |

---

### 1.10.2.9 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Assuming all SQL is ANSI compliant | Vendors differ |
| Ignoring vendor-specific features | Important in real systems |
| Thinking ANSI SQL covers everything | Vendors extend SQL heavily |

### 1.10.3.7 Example Window Function

#### Syntax

```sql
SELECT column_name,
       function_name() OVER (
           PARTITION BY column_name
       )
FROM table_name;
```

#### Example

```sql
SELECT department,
       AVG(salary) OVER (
           PARTITION BY department
       )
FROM employees;
```

## 1.10.4 Vendor Deviations

---

### 1.10.4.1 What are Vendor Deviations

Vendor deviations are:

- Database-specific differences from standard SQL

Each vendor adds:

- Proprietary features
- Custom syntax
- Performance optimizations

---

### 1.10.4.2 Why Vendor Deviations Exist

Database vendors compete by adding:

| Addition | Purpose |
|---|---|
| Faster features | Better performance |
| Specialized functions | Unique capabilities |
| Enterprise tools | Advanced systems |

---

### 1.10.4.3 Real-World Analogy

Think of vendor deviations like:

- Smartphone manufacturers customizing Android

The core system is similar, but:

- Features differ
- Interfaces differ
- Capabilities differ

---

### 1.10.4.4 Example LIMIT Differences

#### MySQL/PostgreSQL

```sql
SELECT *
FROM students
LIMIT 5;
```

#### SQL Server

```sql
SELECT TOP 5 *
FROM students;
```

#### Oracle

```sql
SELECT *
FROM students
FETCH FIRST 5 ROWS ONLY;
```

---

### 1.10.4.5 Example Auto Increment Differences

#### MySQL

```sql
CREATE TABLE students (
    student_id INT AUTO_INCREMENT,
    student_name VARCHAR(100)
);
```

#### PostgreSQL

```sql
CREATE TABLE students (
    student_id SERIAL,
    student_name VARCHAR(100)
);
```

---

### 1.10.4.6 Common Areas with Vendor Differences

| Feature Area | Differences |
|---|---|
| Pagination | LIMIT, TOP, FETCH |
| Auto increment | AUTO_INCREMENT, SERIAL |
| String functions | CONCAT differences |
| Date functions | Vendor-specific syntax |
| JSON support | Different implementations |
| Stored procedures | Different procedural languages |

---

### 1.10.4.7 Why Vendor Deviations Matter

Vendor deviations affect:

| Area | Impact |
|---|---|
| Query portability | Harder migration |
| Developer learning | Need vendor knowledge |
| Cross-platform systems | Compatibility issues |

---

### 1.10.4.8 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Assuming syntax works everywhere | Vendor differences exist |
| Learning proprietary syntax too early | Learn standard SQL first |
| Ignoring portability concerns | Future migration problems |

## 1.10.5 SQL Portability

---

### 1.10.5.1 What is SQL Portability

SQL portability means:

- The ability to move SQL code between database systems

without major modifications.

---

### 1.10.5.2 Why Portability is Important

Portability helps organizations:

| Benefit | Explanation |
|---|---|
| Change vendors easily | Avoid lock-in |
| Migrate systems | Easier upgrades |
| Support multiple databases | Flexible architecture |

---

### 1.10.5.3 Example Portable Query

#### Syntax

```sql
SELECT column_name
FROM table_name
WHERE condition;
```

#### Example

```sql
SELECT employee_name
FROM employees
WHERE salary > 5000;
```

This query works on most SQL databases.

---

### 1.10.5.4 Example Non-Portable Query

#### SQL Server Specific

```sql
SELECT TOP 10 *
FROM employees;
```

This may fail in:

- MySQL
- PostgreSQL

---

### 1.10.5.5 Best Practices for SQL Portability

| Best Practice | Purpose |
|---|---|
| Use ANSI SQL syntax | Better compatibility |
| Avoid proprietary functions | Easier migration |
| Standardize naming | Cleaner systems |
| Test across systems | Validate compatibility |

---

### 1.10.5.6 Real-World Portability Scenario

#### Example Migration

| Old Database | New Database |
|---|---|
| Oracle | PostgreSQL |

Portable SQL reduces:

- Migration complexity
- Rewrite effort
- System downtime

---

### 1.10.5.7 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Using vendor-specific syntax everywhere | Poor portability |
| Ignoring standards | Harder migration |
| Mixing dialects accidentally | Syntax conflicts |

## 1.10.6 Cross-Platform Compatibility

---

### 1.10.6.1 What is Cross-Platform Compatibility

Cross-platform compatibility means:

- SQL systems working consistently across different environments

Examples:

| Environment | Example |
|---|---|
| Operating systems | Windows/Linux |
| Database systems | MySQL/PostgreSQL |
| Cloud platforms | AWS/Azure |

---

### 1.10.6.2 Why Compatibility Matters

Modern systems often use:

- Multiple databases
- Multiple cloud providers
- Hybrid architectures

Compatibility helps systems:

| Benefit | Explanation |
|---|---|
| Integrate easier | Multi-system communication |
| Reduce maintenance | Standard behavior |
| Support scaling | Flexible deployment |

---

### 1.10.6.3 Cross-Platform SQL Challenges

| Challenge | Example |
|---|---|
| Syntax differences | LIMIT vs TOP |
| Datatype differences | VARCHAR behavior |
| Function differences | Date functions |
| Transaction behavior | Vendor implementation |

---

### 1.10.6.4 Example Cross-Platform Query

#### Portable Syntax

```sql
SELECT customer_name
FROM customers
WHERE country = 'Malaysia';
```

This query works in most SQL systems.

---

### 1.10.6.5 Example Compatibility Issue

#### MySQL-Specific Syntax

```sql
SHOW TABLES;
```

This command may not work identically in:

- PostgreSQL
- SQL Server

---

### 1.10.6.6 Strategies for Better Compatibility

| Strategy | Purpose |
|---|---|
| Follow ANSI SQL | Standard behavior |
| Use abstraction layers | Reduce dependency |
| Test across databases | Identify issues |
| Avoid proprietary features | Better portability |

---

### 1.10.6.7 Real-World Enterprise Scenario

Large companies may use:

| System | Database |
|---|---|
| Analytics | Snowflake |
| Web application | PostgreSQL |
| Legacy system | Oracle |

Cross-platform compatibility becomes extremely important.
