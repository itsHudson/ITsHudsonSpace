# 1.5 SQL Execution Flow

---

## 1.5.1 Introduction to SQL Execution Flow

---

### 1.5.1.1 What is SQL Execution Flow

SQL Execution Flow refers to:

- The internal process the database follows when executing an SQL query

When a user writes an SQL statement:

```sql
SELECT *
FROM students;
```

The database does not immediately return results.

Instead:

- The database processes the query through multiple internal stages

---

### 1.5.1.2 Why SQL Execution Flow is Important

Understanding SQL execution flow helps developers:

| Benefit | Explanation |
|---|---|
| Understand query behavior | Learn what happens internally |
| Improve SQL performance | Write efficient queries |
| Debug errors faster | Identify execution issues |
| Understand optimization | Learn how databases improve speed |
| Build scalable systems | Avoid inefficient queries |

---

### 1.5.1.3 Real-World Analogy

Think of SQL execution like:

- Ordering food in a restaurant

| Restaurant Process | SQL Execution Stage |
|---|---|
| Customer places order | SQL query submitted |
| Staff reads order | Parsing |
| Staff checks ingredients | Validation |
| Chef plans cooking | Optimization |
| Chef cooks food | Execution |
| Waiter serves food | Result returning |

---

### 1.5.1.4 Main SQL Execution Stages

Most SQL databases follow these stages:

| Step | Stage |
|---|---|
| Step 1 | Parsing |
| Step 2 | Validation |
| Step 3 | Optimization |
| Step 4 | Execution |
| Step 5 | Result Returning |

---

### 1.5.1.5 Example Query Used Throughout This Section

We will use this query for explanations:

```sql
SELECT student_name
FROM students
WHERE age > 20;
```

---

## 1.5.2 Parsing

---

### 1.5.2.1 What is Parsing

Parsing is:

- The first stage of SQL execution

During parsing:

- The database reads and interprets the SQL query

The parser checks:

- SQL syntax
- Keywords
- Query structure
- Grammar rules

---

### 1.5.2.2 Main Purpose of Parsing

The parser converts:

- Human-readable SQL

into:

- Internal database instructions

---

### 1.5.2.3 Real-World Analogy

Think of parsing like:

- A language teacher checking grammar in a sentence

Example:

| Sentence | Grammar Result |
|---|---|
| "I am a student" | Correct |
| "Am student I" | Incorrect |

The parser does something similar for SQL syntax.

---

### 1.5.2.4 Parsing Example

#### Valid SQL Query

```sql
SELECT student_name
FROM students;
```

The parser recognizes:

| Component | Meaning |
|---|---|
| SELECT | SQL keyword |
| student_name | Column |
| FROM | SQL keyword |
| students | Table |

---

### 1.5.2.5 Syntax Error Example

#### Invalid SQL Query

```sql
SELEC student_name
FROM students;
```

Error:

- SELEC is invalid
- Correct keyword should be SELECT

---

### 1.5.2.6 Common Parsing Errors

| Error Type | Example |
|---|---|
| Misspelled keywords | SELEC |
| Missing commas | Missing separators |
| Missing parentheses | Syntax issue |
| Invalid operators | Wrong symbols |
| Incorrect clause order | FROM before SELECT |

---

### 1.5.2.7 Internal Parsing Output

After parsing:

- The database creates an internal representation of the query

This is often called:

- Parse Tree
- Syntax Tree

---

### 1.5.2.8 Beginner Mistakes During Parsing

| Mistake | Explanation |
|---|---|
| Typing SQL keywords incorrectly | Causes syntax errors |
| Forgetting commas | Invalid query structure |
| Missing quotes around strings | Parsing failure |
| Wrong clause order | SQL grammar violation |

---

## 1.5.3 Validation

---

### 1.5.3.1 What is Validation

After parsing:

- The database validates the query

Validation checks whether:

- Tables exist
- Columns exist
- User permissions are valid
- Datatypes are compatible

---

### 1.5.3.2 Main Purpose of Validation

Validation ensures:

- The query is logically correct

Even if syntax is correct:

- The query may still fail validation

---

### 1.5.3.3 Example of Successful Validation

#### Query

```sql
SELECT student_name
FROM students;
```

Validation checks:

| Check | Result |
|---|---|
| Does students table exist? | Yes |
| Does student_name column exist? | Yes |
| Does user have permission? | Yes |

---

### 1.5.3.4 Validation Failure Example

#### Invalid Query

```sql
SELECT salary
FROM students;
```

Problem:

- salary column does not exist in students

---

### 1.5.3.5 Permission Validation Example

#### Query

```sql
SELECT *
FROM payroll;
```

Possible error:

- User does not have permission

---

### 1.5.3.6 Common Validation Checks

| Validation Type | Purpose |
|---|---|
| Table validation | Ensure table exists |
| Column validation | Ensure column exists |
| Datatype validation | Ensure compatible types |
| Permission validation | Check user access |
| Alias validation | Verify aliases |

---

### 1.5.3.7 Beginner Mistakes During Validation

| Mistake | Explanation |
|---|---|
| Wrong table names | Table not found |
| Wrong column names | Column not found |
| Wrong aliases | Invalid references |
| Ignoring permissions | Access denied |
| Datatype mismatches | Invalid operations |

---

## 1.5.4 Optimization

---

### 1.5.4.1 What is Optimization

Optimization is:

- The stage where the database decides the best way to execute the query

The database optimizer tries to:

- Reduce execution time
- Reduce memory usage
- Improve performance

---

### 1.5.4.2 Why Optimization is Important

The same query can be executed:

- Efficiently
- Inefficiently

Optimization helps choose:

- The fastest execution strategy

---

### 1.5.4.3 Real-World Analogy

Think of optimization like:

- Choosing the fastest route in Google Maps

Different routes may reach the same destination.

However:

- Some routes are faster

The optimizer chooses the best path.

---

### 1.5.4.4 Example Query

```sql
SELECT *
FROM orders
WHERE customer_id = 100;
```

The optimizer decides:

| Decision | Example |
|---|---|
| Use index or full scan | Faster retrieval |
| Join order | Best table sequence |
| Filtering strategy | Reduce processing |

---

### 1.5.4.5 Query Execution Plan

The optimizer creates:

- An execution plan

This plan describes:

- How the database will execute the query

---

### 1.5.4.6 Example Optimization Decisions

| Optimization Decision | Purpose |
|---|---|
| Index usage | Faster searching |
| Join optimization | Efficient table joins |
| Predicate pushdown | Early filtering |
| Parallel execution | Faster large queries |

---

### 1.5.4.7 Index Optimization Example

#### Query

```sql
SELECT *
FROM customers
WHERE customer_id = 100;
```

#### Without Index

- Database scans entire table

#### With Index

- Database jumps directly to matching row

---

### 1.5.4.8 Cost-Based Optimization

Modern databases use:

- Cost-Based Optimizers (CBO)

The optimizer estimates:

| Factor | Example |
|---|---|
| CPU cost | Processing time |
| Disk I/O | Storage access |
| Memory usage | RAM usage |
| Network cost | Distributed systems |

---

### 1.5.4.9 Beginner Mistakes Affecting Optimization

| Mistake | Impact |
|---|---|
| Using SELECT * unnecessarily | Extra processing |
| Missing indexes | Slow queries |
| Poor joins | Large scans |
| Unfiltered queries | Massive datasets |
| Wrong datatypes | Index inefficiency |

---

## 1.5.5 Execution

---

### 1.5.5.1 What is Execution

Execution is:

- The stage where the database actually performs the query operations

The database now:

- Reads data
- Filters rows
- Joins tables
- Sorts results
- Calculates functions

---

### 1.5.5.2 Execution Uses the Query Plan

Execution follows:

- The execution plan created by the optimizer

---

### 1.5.5.3 Example Query Execution

#### Query

```sql
SELECT student_name
FROM students
WHERE age > 20;
```

Execution steps:

| Step | Action |
|---|---|
| Step 1 | Access students table |
| Step 2 | Read rows |
| Step 3 | Check age condition |
| Step 4 | Keep matching rows |
| Step 5 | Return student_name |

---

### 1.5.5.4 Common Execution Operations

| Operation | Purpose |
|---|---|
| Table scan | Read table rows |
| Index scan | Read indexed data |
| Join execution | Combine tables |
| Sorting | ORDER BY processing |
| Aggregation | SUM, AVG, COUNT |

---

### 1.5.5.5 Full Table Scan Example

#### Query

```sql
SELECT *
FROM products;
```

Without filtering:

- Database may scan entire table

---

### 1.5.5.6 Execution Performance Factors

| Factor | Impact |
|---|---|
| Table size | Larger tables slower |
| Indexes | Faster access |
| Query complexity | More processing |
| Hardware | CPU and memory speed |
| Concurrency | Multiple users |

---

### 1.5.5.7 Beginner Mistakes During Execution

| Mistake | Problem |
|---|---|
| Selecting unnecessary columns | More memory usage |
| Large joins without filtering | Slow execution |
| Missing WHERE clauses | Full scans |
| Complex subqueries | Heavy processing |

---

## 1.5.6 Result Returning

---

### 1.5.6.1 What is Result Returning

After execution completes:

- The database returns the query results to the user or application

---

### 1.5.6.2 Result Returning Process

The database:

- Collects final rows
- Formats results
- Sends results to client application

---

### 1.5.6.3 Example Final Result

#### Query

```sql
SELECT student_name
FROM students
WHERE age > 20;
```

#### Result

| student_name |
|---|
| Alice |
| Bob |

---

### 1.5.6.4 Where Results Can Be Returned

| Client | Example |
|---|---|
| SQL terminal | Command line |
| Database tool | DBeaver |
| Backend application | API response |
| Dashboard system | Power BI |
| Web application | Website UI |

---

### 1.5.6.5 Network Transfer Considerations

Large query results may cause:

| Issue | Explanation |
|---|---|
| Slow transfer | Large datasets |
| High memory usage | Huge result sets |
| Network bottlenecks | Distributed systems |

---

### 1.5.6.6 Pagination Example

Instead of returning millions of rows:

- Databases often return smaller pages

#### Example

```sql
SELECT *
FROM students
LIMIT 10;
```

This improves:

- Speed
- User experience
- Memory usage

---

### 1.5.6.7 Beginner Mistakes During Result Returning

| Mistake | Problem |
|---|---|
| Returning too many rows | Slow applications |
| Using SELECT * carelessly | Unnecessary data transfer |
| No pagination | Huge result sets |
| Ignoring network cost | Performance problems |

## 1.5.9 Real-World Performance Example

---

### 1.5.9.1 Slow Query Example

#### Query

```sql
SELECT *
FROM orders;
```

Problem:

- Reads every row
- Reads every column
- Large data transfer

---

### 1.5.9.2 Optimized Query Example

#### Query

```sql
SELECT order_id,
       customer_id
FROM orders
WHERE customer_id = 100;
```

Benefits:

| Improvement | Reason |
|---|---|
| Fewer columns | Less memory |
| Filtering | Fewer rows |
| Possible index usage | Faster retrieval |
