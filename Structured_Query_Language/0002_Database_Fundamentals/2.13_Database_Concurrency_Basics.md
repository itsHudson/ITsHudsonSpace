# 2.13 Database Concurrency Basics

---

# 2.13.1 Introduction to Database Concurrency

Database concurrency refers to:

- The ability of a database system to allow multiple users or multiple transactions to access and modify data simultaneously.

Modern database systems are designed for:

- Thousands of concurrent users
- Continuous transactions
- Real-time processing
- Multi-user applications

Without concurrency control:

- Data corruption may occur
- Transactions may interfere with each other
- Inconsistent results may appear
- Systems become unreliable

---

# 2.13.2 Why Concurrency is Important

Modern systems constantly process simultaneous operations.

Examples:

| System | Concurrent Activity |
|---|---|
| Banking System | Thousands of ATM transactions |
| E-commerce Website | Multiple customers ordering simultaneously |
| Hospital System | Multiple staff updating patient records |
| Airline Reservation System | Concurrent ticket booking |

Concurrency allows databases to:

| Benefit | Explanation |
|---|---|
| Support Multiple Users | Shared access |
| Improve System Efficiency | Parallel operations |
| Increase Throughput | More completed transactions |
| Improve User Experience | Faster responses |

---

# 2.13.3 Real-World Analogy

Think of concurrency like:

- Multiple people using the same library simultaneously.

Rules are needed to prevent problems such as:

- Two people editing the same document incorrectly
- Resources becoming unavailable
- Conflicting operations

Similarly, databases require concurrency control mechanisms.

---

# 2.13.4 Concurrency Problems Without Control

# 2.13.7 Types of Locks

---

## 2.13.7.1 Shared Lock (Read Lock)

A shared lock allows:

- Multiple users to read data simultaneously.

But:

- Writing is blocked.

### Example

```text
User A → READ Account
User B → READ Account
```

Allowed simultaneously.

---

## 2.13.7.2 Exclusive Lock (Write Lock)

An exclusive lock allows:

- Only one transaction to modify data.

Other reads/writes may be restricted.

### Example

```text
User A → UPDATE Account
User B → WAIT
```

---

## 2.13.7.3 Lock Comparison

| Lock Type | Read Allowed | Write Allowed |
|---|---|---|
| Shared Lock | Yes | No |
| Exclusive Lock | Sometimes restricted | Yes |

---

# 2.13.8 Lock Granularity

---

## 2.13.8.1 Definition

Granularity defines:

- The size of data being locked.

---

## 2.13.8.2 Lock Levels

| Lock Level | Description |
|---|---|
| Database Lock | Entire database |
| Table Lock | Entire table |
| Page Lock | Data page |
| Row Lock | Individual row |

---

## 2.13.8.3 Example

### Row Lock

- Only one customer record locked.
- Other rows remain accessible.

### Table Lock

- Entire Customers table locked.
- All users affected.

---

## 2.13.8.4 Trade-Offs

| Smaller Locks | Larger Locks |
|---|---|
| Better concurrency | Less overhead |
| More lock management | Reduced parallelism |

---

# 2.13.9 Blocking

---

## 2.13.9.1 Definition

Blocking occurs when:

- One transaction must wait because another transaction holds a conflicting lock.

---

## 2.13.9.2 Example

### Transaction A

```sql
UPDATE Accounts
SET Balance = 5000
WHERE AccountID = 1001;
```

Transaction A holds exclusive lock.

### Transaction B

```sql
UPDATE Accounts
SET Balance = 7000
WHERE AccountID = 1001;
```

Transaction B must wait.

This waiting is called:

- Blocking

---

## 2.13.9.3 Blocking Flow

```text
Transaction A locks row
        ↓
Transaction B requests same row
        ↓
Transaction B waits
```

---

## 2.13.9.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Waiting State | Resource unavailable |
| Temporary Condition | Ends after lock release |
| Common in High Concurrency | Shared resources |

---

## 2.13.9.5 Advantages

| Advantage | Explanation |
|---|---|
| Prevents Data Corruption | Safe modifications |
| Maintains Consistency | Controlled access |

---

## 2.13.9.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Slower Transactions | Waiting delays |
| Reduced Throughput | Less parallel work |

---

## 2.13.9.7 Real-World Analogy

Think of blocking like:

- Waiting in line for an ATM already being used.

---

# 2.13.10 Deadlocks

---

## 2.13.10.1 Definition

A deadlock occurs when:

- Two or more transactions wait for each other indefinitely.

Each transaction holds a lock needed by another.

---

## 2.13.10.2 Example

### Step-by-Step

### Transaction A

Locks:

- Account 1

Needs:

- Account 2

### Transaction B

Locks:

- Account 2

Needs:

- Account 1

Both wait forever.

---

## 2.13.10.3 Deadlock Diagram

```text
Transaction A → waits for B
Transaction B → waits for A
```

---

## 2.13.10.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Circular Waiting | Transactions depend on each other |
| Infinite Waiting | No progress |
| Automatic Detection Needed | DBMS intervention required |

---

## 2.13.10.5 Deadlock Detection

Most DBMS automatically:

- Detect deadlocks
- Choose victim transaction
- Rollback victim
- Release locks

---

## 2.13.10.6 Deadlock Victim

One transaction becomes:

- Deadlock victim

The victim transaction is rolled back.

---

## 2.13.10.7 Advantages of Detection

| Advantage | Explanation |
|---|---|
| Automatic Recovery | System continues operating |
| Prevents Infinite Waiting | Improves stability |

---

## 2.13.10.8 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Transaction Rollback | Lost work |
| Performance Overhead | Deadlock monitoring |

---

## 2.13.10.9 Real-World Analogy

Think of deadlock like:

- Two cars blocking each other in a narrow road.

Neither can move forward.

# 2.13.13 Common Concurrency Problems

---

## 2.13.13.1 Dirty Read

### Definition

Dirty read occurs when:

- A transaction reads uncommitted data from another transaction.

### Example

Transaction A updates balance:

```sql
UPDATE Accounts
SET Balance = 5000;
```

Not committed yet.

Transaction B reads:

```text
5000
```

Then Transaction A rolls back.

Transaction B used invalid data.

---

## 2.13.13.2 Lost Update

### Definition

Lost update occurs when:

- One transaction overwrites another transaction’s changes.

### Example

### Initial Balance

```text
1000
```

### Transaction A

- Adds 100

### Transaction B

- Subtracts 50

One update accidentally overwrites the other.

---

## 2.13.13.3 Non-repeatable Read

### Definition

Occurs when:

- Same query returns different values during same transaction.

### Example

Transaction A reads salary:

```text
5000
```

Transaction B updates salary:

```text
6000
```

Transaction A reads again:

```text
6000
```

Result changed unexpectedly.

---

## 2.13.13.4 Phantom Read

### Definition

Occurs when:

- New rows appear during repeated query execution.

### Example

Transaction A:

```sql
SELECT *
FROM Orders
WHERE Amount > 1000;
```

Transaction B inserts new matching row.

Transaction A reruns query:

- Additional row appears

---
