# 2.5 Database Objects

---

## 2.5.1 Introduction to Database Objects

Database objects are:

- Named structures or components created inside a database to store, manage, secure, and process data.

Database objects help organize database systems into manageable components.

Almost every database system uses objects such as:

- Tables
- Views
- Indexes
- Stored procedures
- Users
- Roles

Database objects allow databases to become:

| Benefit | Explanation |
|---|---|
| Organized | Structured system design |
| Efficient | Faster data access |
| Secure | Controlled permissions |
| Automated | Reusable logic |
| Scalable | Easier large-system management |

---

## 2.5.2 Why Database Objects are Important

Without database objects:

- Databases would become disorganized
- Performance would decrease
- Security would weaken
- Reusable logic would be difficult
- Maintenance would become complicated

Database objects help separate responsibilities such as:

| Responsibility | Object Type |
|---|---|
| Data Storage | Tables |
| Virtual Data Access | Views |
| Performance Optimization | Indexes |
| Business Logic | Stored procedures |
| Security | Users and roles |
| Storage Management | Tablespaces and partitions |

---

## 2.5.3 Real-World Analogy

Think of a database like:

- A large office building.

Different objects inside the building have different responsibilities.

| Office Building Item | Database Object Equivalent |
|---|---|
| Filing cabinets | Tables |
| Shortcut reports | Views |
| Search directory | Indexes |
| Automated office workflow | Stored procedures |
| Employee access cards | Users and roles |
| Storage rooms | Tablespaces |

---

## 2.5.4 Classification of Database Objects

Database objects are commonly grouped into:

| Category | Purpose |
|---|---|
| Core Objects | Basic data storage and access |
| Programmable Objects | Reusable business logic |
| Sequence Objects | Automatic numbering |

| Storage Objects | Physical storage management |

## 2.5.6 Tables

---

### 2.5.6.1 Definition

A table is:

- A structured object used to store related data in rows and columns.

Tables are the primary storage objects in relational databases.

---

### 2.5.6.2 Example Table

| StudentID | StudentName | Course |
|---|---|---|
| 101 | John | Database Systems |
| 102 | Sarah | Networking |

---

### 2.5.6.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Structured Storage | Organized rows and columns |
| Persistent Data | Data stored permanently |
| Relational Support | Connects with other tables |
| Schema Controlled | Defined structure |

---

### 2.5.6.4 SQL Example

```sql
CREATE TABLE Students (
    StudentID INT PRIMARY KEY,
    StudentName VARCHAR(100),
    Course VARCHAR(100)
);
```

---

### 2.5.6.5 Real-World Usage

| Industry | Example Table |
|---|---|
| Banking | Accounts |
| Hospital | Patients |
| E-commerce | Orders |
| University | Students |

---

### 2.5.6.6 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Creating one table for everything | Causes poor design |
| Using unclear table names | Reduces readability |
| Ignoring primary keys | Causes duplicate issues |

---

## 2.5.7 Views

---

### 2.5.7.1 Definition

A view is:

- A virtual table created from one or more SQL queries.

Views do not usually store data physically.

Instead:

- They display data dynamically from underlying tables.

---

### 2.5.7.2 Real-World Analogy

Think of a view like:

- A filtered window into a database.

The actual data remains in the original tables.

---

### 2.5.7.3 Example

#### Original Table

| StudentID | StudentName | GPA |
|---|---|---|
| 101 | John | 3.80 |
| 102 | Sarah | 2.90 |

#### View

Only students with GPA above 3.00:

| StudentName | GPA |
|---|---|
| John | 3.80 |

---

### 2.5.7.4 SQL Example

```sql
CREATE VIEW HighGPAStudents AS
SELECT StudentName, GPA
FROM Students
WHERE GPA > 3.0;
```

---

### 2.5.7.5 Advantages

| Advantage | Explanation |
|---|---|
| Simplifies Queries | Easier access |
| Improves Security | Restricts sensitive data |
| Reusability | Shared query logic |
| Better Abstraction | Hides complexity |

---

### 2.5.7.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Performance Overhead | Complex views may slow queries |
| Limited Update Support | Some views are read-only |

---

### 2.5.7.7 Types of Views

| Type | Explanation |
|---|---|
| Simple View | Based on one table |
| Complex View | Based on multiple tables |
| Materialized View | Physically stores query results |

---

## 2.5.8 Indexes

---

### 2.5.8.1 Definition

An index is:

- A database object used to improve query performance and search speed.

Indexes work similarly to:

- A book index.

Instead of scanning every page, the system can quickly locate information.

---

### 2.5.8.2 Example Without Index

Suppose a table contains:

- 10 million student records

Without an index:

- The database scans every row.

This is called:

- Full table scan

---

### 2.5.8.3 Example With Index

With an index on StudentID:

- Database jumps directly to required records.

---

### 2.5.8.4 SQL Example

```sql
CREATE INDEX idx_student_name
ON Students(StudentName);
```

---

### 2.5.8.5 Characteristics

| Characteristic | Explanation |
|---|---|
| Performance Optimization | Faster searching |
| Additional Storage | Requires extra space |
| Sorted Structure | Organized lookup |
| Query Acceleration | Faster retrieval |

---

### 2.5.8.6 Advantages

| Advantage | Explanation |
|---|---|
| Faster Queries | Improved search performance |
| Faster Sorting | Better ORDER BY operations |
| Faster Filtering | Better WHERE clause performance |

---

### 2.5.8.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Additional Storage | Extra disk usage |
| Slower Inserts/Updates | Index maintenance overhead |

---

### 2.5.8.8 Common Index Types

| Index Type | Purpose |
|---|---|
| Primary Index | Primary key optimization |
| Unique Index | Prevent duplicates |
| Composite Index | Multiple columns |
| Full-Text Index | Text searching |

## 2.5.10 Stored Procedures

---

### 2.5.10.1 Definition

A stored procedure is:

- A precompiled collection of SQL statements stored inside the database.

Stored procedures can:

- Accept parameters
- Execute logic
- Return results

---

### 2.5.10.2 Real-World Analogy

Think of a stored procedure like:

- A reusable machine operation.

Instead of manually repeating steps:

- One button performs everything automatically.

---

### 2.5.10.3 SQL Example

```sql
CREATE PROCEDURE GetStudents
AS
BEGIN
    SELECT * FROM Students;
END;
```

---

### 2.5.10.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Reusable Logic | Repeated execution |
| Stored in Database | Centralized processing |
| Parameter Support | Dynamic input |
| Security Control | Controlled access |

---

### 2.5.10.5 Advantages

| Advantage | Explanation |
|---|---|
| Reusability | Avoid repeated code |
| Better Security | Restricted direct table access |
| Performance | Precompiled execution |
| Centralized Logic | Easier maintenance |

---

### 2.5.10.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Vendor Differences | Syntax varies by DBMS |
| Debugging Complexity | Harder troubleshooting |

---

## 2.5.11 Functions

---

### 2.5.11.1 Definition

A function is:

- A reusable database object that performs operations and returns a value.

Functions are commonly used in:

- Calculations
- Transformations
- Data formatting

---

### 2.5.11.2 SQL Example

```sql
CREATE FUNCTION CalculateBonus(
    @salary DECIMAL(10,2)
)
RETURNS DECIMAL(10,2)
AS
BEGIN
    RETURN @salary * 0.10;
END;
```

---

### 2.5.11.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Returns Value | Produces output |
| Reusable Logic | Shared calculations |
| Parameter Support | Accepts input |

---

### 2.5.11.4 Functions vs Stored Procedures

| Feature | Function | Stored Procedure |
|---|---|---|
| Returns Value | Yes | Optional |
| Used Inside Queries | Yes | Usually no |
| Main Purpose | Calculations | Operations |

---

## 2.5.12 Triggers

---

### 2.5.12.1 Definition

A trigger is:

- A database object that automatically executes when certain database events occur.

Triggers respond to events such as:

- INSERT
- UPDATE
- DELETE

---

### 2.5.12.2 Real-World Analogy

Think of a trigger like:

- A motion sensor light.

When movement occurs:

- The light activates automatically.

Similarly:

- Triggers automatically execute when events happen.

---

### 2.5.12.3 SQL Example

```sql
CREATE TRIGGER StudentInsertTrigger
ON Students
AFTER INSERT
AS
BEGIN
    PRINT 'New student added';
END;
```

---

### 2.5.12.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Event-Driven | Automatic execution |
| Hidden Execution | Runs behind the scenes |
| Rule Enforcement | Maintains consistency |

---

### 2.5.12.5 Advantages

| Advantage | Explanation |
|---|---|
| Automation | Automatic operations |
| Auditing | Tracks changes |
| Integrity Enforcement | Business rules |

---

### 2.5.12.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Hard Debugging | Hidden execution |
| Performance Overhead | Additional processing |

## 2.5.14 Sequences

---

### 2.5.14.1 Definition

A sequence is:

- A database object that generates sequential numeric values automatically.

Sequences are commonly used for:

- Primary keys
- Invoice numbers
- Order numbers

---

### 2.5.14.2 SQL Example

```sql
CREATE SEQUENCE StudentSeq
START WITH 1
INCREMENT BY 1;
```

---

### 2.5.14.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Automatic Number Generation | Sequential values |
| Reusable | Multiple tables possible |
| Independent Object | Separate from tables |

---

### 2.5.14.4 Advantages

| Advantage | Explanation |
|---|---|
| Prevents Duplicate IDs | Unique generation |
| Simplifies Inserts | Automatic numbering |

---

## 2.5.15 Auto Increment

---

### 2.5.15.1 Definition

Auto increment is:

- A feature that automatically increases numeric values during inserts.

---

### 2.5.15.2 SQL Example

```sql
CREATE TABLE Students (
    StudentID INT AUTO_INCREMENT PRIMARY KEY,
    StudentName VARCHAR(100)
);
```

---

### 2.5.15.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Automatic ID Generation | Sequential IDs |
| Table-Based | Built into column |
| Simplified Inserts | Less manual input |

---

### 2.5.15.4 Sequences vs Auto Increment

| Feature | Sequence | Auto Increment |
|---|---|---|
| Independent Object | Yes | No |
| Reusable Across Tables | Yes | Usually no |
| Flexibility | Higher | Simpler |

## 2.5.22 Partitions

---

### 2.5.22.1 Definition

Partitioning is:

- Splitting large tables into smaller manageable pieces.

Each partition contains part of the data.

---

### 2.5.22.2 Example

Large sales table partitioned by year:

| Partition | Data |
|---|---|
| 2024 | Sales for 2024 |
| 2025 | Sales for 2025 |
| 2026 | Sales for 2026 |

---

### 2.5.22.3 Advantages

| Advantage | Explanation |
|---|---|
| Better Performance | Smaller scans |
| Easier Maintenance | Manage sections separately |
| Improved Scalability | Large-table optimization |

---

### 2.5.22.4 Types of Partitioning

| Type | Explanation |
|---|---|
| Range Partitioning | Based on value ranges |
| List Partitioning | Based on predefined lists |
| Hash Partitioning | Based on hashing |
