# 2.7 Database Keys

---

## 2.7.1 Introduction to Database Keys

Database keys are:

- Special columns or combinations of columns used to identify, connect, and enforce integrity within database tables.

Keys are one of the most important concepts in relational databases.

Without keys:

- Duplicate data increases
- Relationships break
- Data integrity becomes difficult
- Queries become unreliable
- Database normalization becomes impossible

Database keys help databases:

| Purpose | Explanation |
|---|---|
| Identify Records | Uniquely distinguish rows |
| Connect Tables | Create relationships |
| Prevent Duplicates | Enforce uniqueness |
| Maintain Integrity | Ensure accurate relationships |
| Improve Querying | Faster searches and joins |

---

## 2.7.2 Why Database Keys are Important

Modern systems require:

- Accurate data
- Connected tables
- Fast retrieval
- Reliable relationships

Database keys help systems avoid problems such as:

| Problem | Example |
|---|---|
| Duplicate users | Same customer entered multiple times |
| Broken relationships | Orders linked to non-existing customers |
| Inconsistent records | Multiple conflicting records |
| Slow searching | No efficient lookup mechanism |

---

## 2.7.3 Real-World Analogy

Think of database keys like:

- Identification systems in real life.

Examples:

| Real-World Identifier | Database Equivalent |
|---|---|
| Passport Number | Primary Key |
| Student ID | Unique Key |
| Car Plate Number | Candidate Key |
| Employee Badge ID | Surrogate Key |

Keys uniquely identify entities.

---

## 2.7.4 Classification of Database Keys

Database keys are commonly categorized as:

| Key Type | Main Purpose |
|---|---|
| Primary Key | Main unique identifier |
| Foreign Key | Relationship connector |
| Candidate Key | Possible primary key |
| Composite Key | Multiple-column key |
| Alternate Key | Candidate key not selected |
| Surrogate Key | Artificial generated key |
| Natural Key | Real-world meaningful key |
| Unique Key | Enforces uniqueness |

---

## 2.7.5 Primary Key

---

### 2.7.5.1 Definition

A primary key is:

- A column or combination of columns that uniquely identifies each row in a table.

A primary key must:

- Be unique
- Not contain NULL values

Every table should ideally have:

- One primary key

---

### 2.7.5.2 Example

#### Students Table

| StudentID | StudentName |
|---|---|
| 101 | John |
| 102 | Sarah |

StudentID is the primary key.

---

### 2.7.5.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Unique | No duplicates allowed |
| NOT NULL | Must contain value |
| One Primary Key Per Table | Only one primary key definition |
| Stable | Should rarely change |

---

### 2.7.5.4 SQL Example

```sql
CREATE TABLE Students (
    StudentID INT PRIMARY KEY,
    StudentName VARCHAR(100)
);
```

---

### 2.7.5.5 Advantages

| Advantage | Explanation |
|---|---|
| Prevents Duplicate Records | Unique identification |
| Supports Relationships | Used by foreign keys |
| Faster Searching | Often indexed automatically |

---

### 2.7.5.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Poor Key Design Causes Problems | Changing keys difficult |
| Large Keys Affect Performance | Bigger indexes |

---

### 2.7.5.7 Real-World Examples

| Example | Explanation |
|---|---|
| Student ID | Unique student identification |
| Passport Number | Unique citizen identification |
| Employee ID | Unique employee identification |

---

### 2.7.5.8 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Using non-unique values | Breaks integrity |
| Allowing NULL primary keys | Invalid design |
| Using frequently changing data | Causes update problems |

---

## 2.7.6 Foreign Key

---

### 2.7.6.1 Definition

A foreign key is:

- A column that references the primary key of another table.

Foreign keys create:

- Relationships between tables

---

### 2.7.6.2 Example

#### Departments Table

| DepartmentID | DepartmentName |
|---|---|
| D01 | IT |

#### Employees Table

| EmployeeID | EmployeeName | DepartmentID |
|---|---|---|
| 101 | John | D01 |

DepartmentID in Employees is a foreign key.

---

### 2.7.6.3 Structure Diagram

```text
Departments
     ↑
Employees
```

---

### 2.7.6.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Creates Relationships | Connects tables |
| Enforces Referential Integrity | Prevents invalid references |
| May Contain Duplicates | Multiple rows can reference same parent |

---

### 2.7.6.5 SQL Example

```sql
CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    EmployeeName VARCHAR(100),
    DepartmentID VARCHAR(10),
    FOREIGN KEY (DepartmentID)
    REFERENCES Departments(DepartmentID)
);
```

---

### 2.7.6.6 Advantages

| Advantage | Explanation |
|---|---|
| Maintains Data Consistency | Valid relationships |
| Prevents Orphan Records | Child must match parent |
| Supports Relational Design | Connected tables |

---

### 2.7.6.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Additional Validation Overhead | Integrity checks |
| Complex Cascading Operations | Deletes/updates affect relationships |

---

## 2.7.7 Candidate Key

---

### 2.7.7.1 Definition

A candidate key is:

- A column or combination of columns that can uniquely identify rows and is eligible to become the primary key.

A table may have:

- Multiple candidate keys

Only one is selected as:

- Primary key

---

### 2.7.7.2 Example

#### Employees Table

| EmployeeID | Email | PassportNumber |
|---|---|---|
| 101 | john@email.com | P88991 |

Possible candidate keys:

- EmployeeID
- Email
- PassportNumber

All uniquely identify employees.

---

### 2.7.7.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Unique | No duplicates |
| Minimal | No unnecessary columns |
| Eligible for Primary Key | Possible identifier |

---

### 2.7.7.4 Candidate Key vs Primary Key

| Feature | Candidate Key | Primary Key |
|---|---|---|
| Can Uniquely Identify | Yes | Yes |
| Selected as Main Key | Not always | Yes |
| Quantity Per Table | Multiple possible | Only one |

---

### 2.7.7.5 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Thinking candidate key equals primary key | Primary key is selected candidate key |
| Using unstable candidate keys | Causes future problems |

---

## 2.7.8 Composite Key

---

### 2.7.8.1 Definition

A composite key is:

- A key made from two or more columns combined together to uniquely identify a record.

No single column alone is unique.

---

### 2.7.8.2 Example

#### Enrollment Table

| StudentID | CourseID |
|---|---|
| 101 | C01 |
| 101 | C02 |

Neither StudentID nor CourseID alone is unique.

Together:

- StudentID + CourseID

form a composite key.

---

### 2.7.8.3 SQL Example

```sql
CREATE TABLE Enrollment (
    StudentID INT,
    CourseID VARCHAR(10),
    PRIMARY KEY (StudentID, CourseID)
);
```

---

### 2.7.8.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Multiple Columns | Combined uniqueness |
| Common in Junction Tables | Many-to-many relationships |
| Relationship-Oriented | Often connects entities |

---

### 2.7.8.5 Advantages

| Advantage | Explanation |
|---|---|
| Better Relationship Modeling | Complex systems |
| Avoids Artificial Keys | Uses real relationships |

---

### 2.7.8.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Larger Indexes | More storage |
| More Complex Queries | Multiple columns required |

---

## 2.7.9 Alternate Key

---

### 2.7.9.1 Definition

An alternate key is:

- A candidate key that was not selected as the primary key.

---

### 2.7.9.2 Example

#### Employees Table

| EmployeeID | Email |
|---|---|
| 101 | john@email.com |

Suppose:

- EmployeeID = Primary Key
- Email = Alternate Key

Email still remains unique.

---

### 2.7.9.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Unique | No duplicates |
| Not Primary Key | Secondary candidate |
| Often Enforced Using UNIQUE Constraint | Integrity support |

---

### 2.7.9.4 Advantages

| Advantage | Explanation |
|---|---|
| Additional Uniqueness Enforcement | Prevent duplicates |
| Flexible Design | Multiple identifiers |

---

## 2.7.10 Surrogate Key

---

### 2.7.10.1 Definition

A surrogate key is:

- An artificial system-generated key with no business meaning.

Usually generated automatically.

---

### 2.7.10.2 Example

| CustomerID | CustomerName |
|---|---|
| 1001 | John |

CustomerID may simply be auto-generated.

---

### 2.7.10.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Artificial Value | No real-world meaning |
| System Generated | Auto increment or sequence |
| Stable | Rarely changes |

---

### 2.7.10.4 Advantages

| Advantage | Explanation |
|---|---|
| Simpler Relationships | Easier joins |
| Better Performance | Smaller indexes |
| Stable Keys | Business data changes safely |

---

### 2.7.10.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| No Business Meaning | Human-unfriendly |
| Additional Column Required | Extra storage |

---

### 2.7.10.6 Real-World Examples

| Example | Explanation |
|---|---|
| Auto-generated IDs | Database-generated values |
| UUIDs | Globally unique identifiers |

---

## 2.7.11 Natural Key

---

### 2.7.11.1 Definition

A natural key is:

- A key derived from real-world business data.

The value already exists naturally.

---

### 2.7.11.2 Examples

| Natural Key | Meaning |
|---|---|
| Passport Number | Government identifier |
| Email Address | User identity |
| National ID | Citizen identifier |

---

### 2.7.11.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Business Meaning | Real-world value |
| Existing Data | Already available |
| Human Readable | Easier understanding |

---

### 2.7.11.4 Advantages

| Advantage | Explanation |
|---|---|
| Meaningful | Easier interpretation |
| No Extra Column Needed | Existing data reused |

---

### 2.7.11.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| May Change | Business data updates |
| Larger Size | Bigger indexes |
| Privacy Risks | Sensitive identifiers |

---

### 2.7.11.6 Natural Key vs Surrogate Key

| Feature | Natural Key | Surrogate Key |
|---|---|---|
| Business Meaning | Yes | No |
| System Generated | No | Yes |
| Stability | May change | Stable |
| Human Readable | Yes | Usually no |

---

## 2.7.12 Unique Key

---

### 2.7.12.1 Definition

A unique key is:

- A constraint that ensures all values in a column are unique.

Unlike primary keys:

- Some DBMS allow NULL values in unique keys.

---

### 2.7.12.2 Example

#### Users Table

| UserID | Email |
|---|---|
| 101 | john@email.com |
| 102 | sarah@email.com |

Emails must remain unique.

---

### 2.7.12.3 SQL Example

```sql
CREATE TABLE Users (
    UserID INT PRIMARY KEY,
    Email VARCHAR(100) UNIQUE
);
```

---

### 2.7.12.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Prevents Duplicates | Unique values enforced |
| Multiple Unique Keys Allowed | Unlike primary key |
| Supports Candidate Keys | Integrity enforcement |

---

### 2.7.12.5 Unique Key vs Primary Key

| Feature | Primary Key | Unique Key |
|---|---|---|
| NULL Allowed | No | Sometimes |
| Number Per Table | One | Multiple |
| Main Identifier | Yes | Usually secondary |

---

### 2.7.13.1 Key Hierarchy

```text
Candidate Keys
     ↓
Primary Key
     ↓
Foreign Keys
```

---

### 2.7.13.2 Relationship Flow Example

#### Customers Table

| CustomerID | CustomerName |
|---|---|
| 1001 | John |

CustomerID = Primary Key

#### Orders Table

| OrderID | CustomerID |
|---|---|
| O01 | 1001 |

CustomerID = Foreign Key

## 2.7.15 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Confusing primary and foreign keys | Primary identifies, foreign references |
| Using unstable natural keys | Causes maintenance problems |
| Ignoring unique constraints | Allows duplicate data |
| Overusing composite keys | Creates complex queries |
| Not indexing important keys | Reduces performance |

---

## 2.7.16 Key Design Best Practices

---

### 2.7.16.1 Use Stable Primary Keys

Primary keys should:

- Rarely change

---

### 2.7.16.2 Keep Keys Small

Smaller keys improve:

- Index performance
- Query speed

---

### 2.7.16.3 Use Foreign Keys Properly

Foreign keys maintain:

- Referential integrity

---

### 2.7.16.4 Avoid Sensitive Natural Keys

Avoid using:

- National IDs
- Personal data

as main identifiers when possible.

---

### 2.7.16.5 Use Composite Keys Carefully

Composite keys are useful but can:

- Increase complexity
