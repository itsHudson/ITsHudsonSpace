# 2.14 Database Logging

---

# 2.14.1 Introduction to Database Logging

Database logging refers to:

- The process of recording database operations, changes, transactions, and system activities into log records or log files.

Logging is one of the most important mechanisms in database systems because it helps provide:

- Recovery
- Durability
- Auditing
- Crash protection
- Transaction consistency
- Security tracking

Without logging:

- Databases could not recover safely from crashes
- Transactions could become inconsistent
- Data corruption risks would increase
- Audit tracking would be impossible

---

# 2.14.2 Why Database Logging is Important

Modern database systems continuously process:

- Transactions
- Updates
- Inserts
- Deletes
- Security operations

Database logging helps ensure:

| Purpose | Explanation |
|---|---|
| Crash Recovery | Restore failed systems |
| Transaction Recovery | Undo/redo operations |
| Durability | Protect committed data |
| Auditing | Track user activity |
| Compliance | Meet legal/security requirements |
| Replication Support | Synchronize databases |

---

# 2.14.3 Real-World Analogy

Think of database logging like:

- A flight recorder (black box) in an airplane.

Even if a system crashes:

- The recorded history remains available.

This allows investigators or recovery systems to:

- Restore operations
- Analyze problems
- Recover data

---

# 2.14.4 Main Types of Database Logs

The major database logging mechanisms include:

| Log Type | Main Purpose |
|---|---|
| Write-Ahead Logging (WAL) | Protect changes before writing data |
| Transaction Logs | Record transaction operations |
| Redo Logs | Reapply committed changes |
| Undo Logs | Reverse uncommitted changes |
| Audit Logs | Track security and activity |

---

# 2.14.5 Write-Ahead Logging (WAL)

---

## 2.14.5.1 Definition

Write-Ahead Logging (WAL) is:

- A logging technique where changes are written to the log before the actual database data files are updated.

This is a core principle used in many modern DBMS.

---

## 2.14.5.2 Core Rule of WAL

The WAL rule is:

- Log first, data later.

---

## 2.14.5.3 Why WAL is Important

Suppose a system crashes during data update.

Without WAL:

- Database may become inconsistent.

With WAL:

- Recovery system can replay logged operations safely.

---

## 2.14.5.4 Step-by-Step WAL Workflow

### Example Transaction

```sql
UPDATE Accounts
SET Balance = 5000
WHERE AccountID = 1001;
```

### WAL Process

#### Step 1 — Change Recorded in Log

- Log entry written first.

#### Step 2 — Log Saved to Disk

- Log safely stored.

#### Step 3 — Actual Data Page Updated

- Database file updated afterward.

---

## 2.14.5.5 Characteristics

| Characteristic | Explanation |
|---|---|
| Log-First Strategy | Logs written before data |
| Recovery-Oriented | Crash protection |
| Durability Support | ACID durability |

---

## 2.14.5.6 Advantages

| Advantage | Explanation |
|---|---|
| Strong Crash Recovery | Reliable restoration |
| Prevents Data Corruption | Safer writes |
| Supports Durability | Permanent committed data |

---

## 2.14.5.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Additional Disk Writes | More I/O overhead |
| Log Management Complexity | Requires maintenance |

---

## 2.14.5.8 Real-World Analogy

Think of WAL like:

- Writing an official receipt before delivering goods.

Even if delivery fails:

- The official record still exists.

# 2.14.7 Redo Logs

---

## 2.14.7.1 Definition

Redo logs are:

- Logs used to reapply committed changes after system failure.

Redo logs ensure:

- Committed transactions are not lost.

---

## 2.14.7.2 Purpose of Redo Logs

Suppose:

- Transaction committed
- System crashes before data fully written

Redo logs help:

- Replay committed changes

---

### Step-by-Step

- Money transfer completed.

#### Step 2 — Crash Occurs

- Data file update incomplete.

#### Step 3 — Recovery Process Reads Redo Logs

- Committed changes replayed.

| Recovery-Oriented | Restore committed work |
| Commit Protection | Durability support |
| Sequential Logging | Ordered replay |

## 2.14.7.5 Advantages

| Advantage | Explanation |
|---|---|
| Prevents Data Loss | Recover committed transactions |
| Supports Durability | Reliable systems |

---

## 2.14.7.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Extra Storage | Additional logs |
| Recovery Overhead | Replay processing |

---

## 2.14.7.7 Real-World Analogy

- Replaying CCTV footage to reconstruct completed events.

---

Undo logs are:

- Logs used to reverse incomplete or uncommitted transactions.

Undo logs restore:

- Previous data state.

- Rollback occurs
- Crashes interrupt operations

### Step 3 — Undo Log Used

## 2.14.8.6 Disadvantages

## 2.14.8.7 Real-World Analogy

- Changes can be reversed.

# 2.14.9 Redo Logs vs Undo Logs

| Feature | Redo Logs | Undo Logs |
|---|---|---|
| Purpose | Reapply changes | Reverse changes |
| Used For | Committed transactions | Uncommitted transactions |
| Recovery Goal | Preserve committed work | Remove incomplete work |
| Direction | Forward recovery | Backward recovery |

# 2.14.11 Database Recovery Process

---

## 2.14.11.1 Crash Recovery Workflow

### Step-by-Step

#### Step 1 — System Crash Occurs

- Database unexpectedly stops.

#### Step 2 — Database Restarts

- Recovery process begins.

#### Step 3 — Transaction Logs Analyzed

- DBMS reads transaction history.

#### Step 4 — Undo Logs Reverse Incomplete Transactions

- Failed or partial changes removed.

#### Step 5 — Redo Logs Replay Committed Transactions

- Successful committed changes restored.

#### Step 6 — Consistent Database Restored

- Database returns to stable state.

---

## 2.14.11.2 Recovery Flow Diagram

```text
Crash Occurs
      ↓
Read Logs
      ↓
Undo Incomplete Transactions
      ↓
Redo Committed Transactions
      ↓
Consistent Database Restored
```

- Additional storage usage

---

## 2.14.14.2 Trade-Off

| Strong Logging | Weak Logging |
|---|---|

# 2.14.15 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Thinking logs are optional | Critical for recovery |
| Confusing redo and undo logs | One reapplies, one reverses |
| Ignoring audit logs | Important for security |
| Assuming commits instantly update all files | WAL may delay physical writes |
| Forgetting log maintenance | Logs can grow very large |
