# 2.2 Database Architecture

---

## 2.2.1 Introduction to Database Architecture

Database architecture refers to:

- The overall design and structure that defines how database components interact with users, applications, servers, and storage systems.

Database architecture determines:

- How users access data
- How applications communicate with databases
- How data flows through systems
- How scalable the system is
- How secure the system is
- How performance is optimized

Database architecture is extremely important because poor architecture can cause:

- Slow system performance
- Security issues
- System crashes
- Scalability limitations
- Maintenance difficulties

Good database architecture helps systems become:

| Benefit | Explanation |
|---|---|
| Scalable | Can handle increasing users and data |
| Secure | Protects sensitive information |
| Reliable | Stable and fault tolerant |
| Maintainable | Easier to manage and upgrade |
| Performant | Faster response times |
| Flexible | Easier to extend and modify |

---

## 2.2.2 Why Database Architecture is Important

---

### 2.2.2.1 Real-World Importance

Modern systems may have:

- Millions of users
- Billions of records
- Thousands of simultaneous transactions
- Global access requirements

Without proper architecture:

- Systems become unstable
- Databases become bottlenecks
- Maintenance becomes difficult
- Scaling becomes expensive

---

### 2.2.2.2 Real-World Analogy

Think of database architecture like:

- Designing a city transportation system.

A well-designed city has:

- Organized roads
- Traffic control
- Efficient transportation
- Backup routes
- Scalability for future growth

Similarly, database architecture organizes:

- Data movement
- System communication
- User access
- Workload distribution

---

## 2.2.3 Basic Database Architecture

Basic database architecture defines:

- How users, applications, and databases are connected at a high level.

The most common architectures are:

| Architecture | Description |
|---|---|
| Single-tier | Everything on one system |
| Two-tier | Client communicates directly with database |
| Three-tier | Application server placed between client and database |

---

## 2.2.4 Single-Tier Architecture

---

### 2.2.4.1 Definition

Single-tier architecture is:

- A database architecture where the user, application, and database all exist on the same machine or system.

Everything operates locally.

---

### 2.2.4.2 Structure

```text
User
  ↓
Application
  ↓
Database
(All inside one system)
```

---

### 2.2.4.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Local Processing | Everything runs locally |
| Simple Setup | Minimal configuration |
| Small Scale | Suitable for personal systems |
| Low Network Dependency | No remote communication required |

---

### 2.2.4.4 Real-World Example

Examples include:

| Example | Explanation |
|---|---|
| Personal desktop applications | Local accounting software |
| Offline mobile apps | Local app databases |
| Lightweight tools | Embedded systems |

SQLite is commonly used in single-tier systems.

---

### 2.2.4.5 Advantages

| Advantage | Explanation |
|---|---|
| Easy Setup | Very simple deployment |
| Fast Local Access | No network delay |
| Low Cost | Minimal infrastructure |
| Simple Maintenance | Easier management |

---

### 2.2.4.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Poor Scalability | Difficult for large systems |
| Limited Multi-user Support | Not ideal for concurrent users |
| Security Limitations | Everything stored locally |
| Single Device Dependency | Device failure affects entire system |

---

### 2.2.4.7 Beginner Mistake

Many beginners assume:

- All databases work locally on one computer.

In real enterprise systems:

- Databases are usually separated from applications
- Multiple servers are involved
- Networks are heavily used

---

## 2.2.5 Two-Tier Architecture

---

### 2.2.5.1 Definition

Two-tier architecture is:

- A client-server architecture where the client application communicates directly with the database server.

The client handles:

- User interface
- Some business logic

The database server handles:

- Data storage
- Query processing

---

### 2.2.5.2 Structure

```text
Client Application
        ↓
   Database Server
```

---

### 2.2.5.3 Components

| Component | Responsibility |
|---|---|
| Client | User interaction |
| Database Server | Data management |

---

### 2.2.5.4 Real-World Example

Examples include:

| System | Example |
|---|---|
| Desktop ERP | Employee management system |
| University systems | Direct DB connections |
| Small business software | POS systems |

---

### 2.2.5.5 Advantages

| Advantage | Explanation |
|---|---|
| Better Multi-user Support | Multiple clients can connect |
| Centralized Data | Easier data consistency |
| Easier Backup | Database server centralized |
| Better Resource Sharing | Shared data environment |

---

### 2.2.5.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Tight Coupling | Client depends heavily on database |
| Security Risks | Clients connect directly to database |
| Scalability Limitations | Harder to scale large systems |
| Maintenance Complexity | Updating many clients may be difficult |

---

### 2.2.5.7 Real-World Workflow

#### Banking Desktop System

Step-by-step flow:

- Employee logs into banking software
- Client application sends SQL queries
- Database processes request
- Results returned to client
- Employee sees customer data

---

## 2.2.6 Three-Tier Architecture

---

### 2.2.6.1 Definition

Three-tier architecture is:

- A system architecture where an application server sits between the client and database server.

This is the most common modern enterprise architecture.

---

### 2.2.6.2 Structure

```text
Client
   ↓
Application Server
   ↓
Database Server
```

---

### 2.2.6.3 Components

| Layer | Responsibility |
|---|---|
| Presentation Layer | User interface |
| Application Layer | Business logic |
| Database Layer | Data storage |

---

### 2.2.6.4 Real-World Analogy

Think of a restaurant:

| Role | System Equivalent |
|---|---|
| Customer | Client |
| Waiter | Application server |
| Kitchen | Database |

The customer does not directly access the kitchen.

Similarly:

- Clients do not directly access the database.

---

### 2.2.6.5 Advantages

| Advantage | Explanation |
|---|---|
| Better Security | Database hidden from clients |
| Better Scalability | Easier to expand layers |
| Easier Maintenance | Layers separated |
| Better Performance | Application server handles logic |
| Reusability | Services shared across systems |

---

### 2.2.6.6 Disadvantages

| Disadvantage | Explanation |
|---|---|
| More Complex | More infrastructure required |
| Higher Cost | Additional servers needed |
| More Configuration | Deployment complexity |

---

### 2.2.6.7 Real-World Examples

| System | Architecture |
|---|---|
| Banking websites | Three-tier |
| E-commerce platforms | Three-tier |
| Social media systems | Three-tier |
| Hospital systems | Three-tier |

---

## 2.2.7 Client-Server Architecture

---

### 2.2.7.1 Definition

Client-server architecture is:

- A computing model where clients request services and servers provide services.

In databases:

- Clients request data
- Database servers process requests

---

### 2.2.7.2 Structure

```text
Client → Server → Database
```

---

### 2.2.7.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Centralized Services | Servers manage resources |
| Multiple Clients | Many users supported |
| Network-Based | Communication over networks |
| Shared Resources | Centralized data access |

---

### 2.2.7.4 Examples

| Example | Description |
|---|---|
| Web browsers accessing websites | Client-server |
| Mobile apps accessing APIs | Client-server |
| ATM systems | Client-server |

---

## 2.2.8 Multi-Tier Architecture

---

### 2.2.8.1 Definition

Multi-tier architecture is:

- An extension of three-tier architecture with multiple intermediate layers.

Additional layers may include:

- API gateways
- Authentication services
- Caching layers
- Analytics services

---

### 2.2.8.2 Structure

```text
Client
   ↓
Web Server
   ↓
Application Server
   ↓
API Services
   ↓
Database
```

---

### 2.2.8.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Highly Modular | Components separated |
| Highly Scalable | Easier expansion |
| Better Security | Multiple protection layers |
| Complex Infrastructure | Many interconnected services |

---

### 2.2.8.4 Real-World Usage

Large companies use multi-tier systems:

| Company | Usage |
|---|---|
| Amazon | E-commerce systems |
| Netflix | Streaming systems |
| Google | Global service platforms |

---

## 2.2.9 Service-Oriented Architecture (SOA)

---

### 2.2.9.1 Definition

Service-Oriented Architecture (SOA) is:

- An architecture where business functions are separated into reusable services.

Each service performs a specific business task.

---

### 2.2.9.2 Examples of Services

| Service | Responsibility |
|---|---|
| Payment Service | Handles payments |
| User Service | Manages accounts |
| Inventory Service | Tracks inventory |
| Notification Service | Sends messages |

---

### 2.2.9.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Reusable Services | Services reused across systems |
| Loose Coupling | Services independent |
| Network Communication | Services communicate remotely |
| Enterprise Integration | Systems connected together |

---

### 2.2.9.4 Advantages

| Advantage | Explanation |
|---|---|
| Reusability | Shared business services |
| Flexibility | Easier updates |
| Integration | Connects multiple systems |
| Scalability | Independent service scaling |

---

### 2.2.9.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Complexity | Service management difficult |
| Network Overhead | More communication cost |
| Monitoring Challenges | Many services to monitor |

---

### 2.2.9.6 Real-World Example

An airline system may separate:

- Booking service
- Payment service
- Customer service
- Flight scheduling service

Each service may use its own database.

---

## 2.2.10 Microservices Database Architecture

---

### 2.2.10.1 Definition

Microservices database architecture is:

- An architecture where small independent services manage their own databases.

Each microservice:

- Owns its data
- Operates independently
- Can use different database technologies

---

### 2.2.10.2 Structure

```text
User Service → User Database
Order Service → Order Database
Payment Service → Payment Database
```

---

### 2.2.10.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Independent Services | Services isolated |
| Independent Databases | Each service controls data |
| High Scalability | Services scale separately |
| Technology Flexibility | Different DBs allowed |

---

### 2.2.10.4 Advantages

| Advantage | Explanation |
|---|---|
| Independent Scaling | Services scale separately |
| Fault Isolation | One service failure isolated |
| Faster Development | Teams work independently |
| Flexibility | Technology freedom |

---

### 2.2.10.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Complex Data Consistency | Harder transaction management |
| Distributed Complexity | Many moving parts |
| Monitoring Difficulty | Complex observability |
| Data Duplication | Possible redundancy |

---

### 2.2.10.6 Real-World Examples

| Company | Usage |
|---|---|
| Netflix | Streaming microservices |
| Uber | Ride management services |
| Amazon | Large-scale e-commerce services |

---

## 2.2.11 Distributed Database Architecture

Distributed database architecture defines:

- How multiple database servers cooperate and share resources.

Three major architectures are:

| Architecture | Description |
|---|---|
| Shared Memory | Servers share memory |
| Shared Disk | Servers share storage disks |
| Shared Nothing | Servers fully independent |

---

## 2.2.12 Shared Memory Architecture

---

### 2.2.12.1 Definition

Shared memory architecture is:

- A distributed architecture where processors share the same memory and storage.

---

### 2.2.12.2 Structure

```text
CPU 1 ─┐
CPU 2 ─┼→ Shared Memory → Shared Disk
CPU 3 ─┘
```

---

### 2.2.12.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Shared RAM | Common memory pool |
| Shared Storage | Same disk access |
| Fast Communication | Internal memory communication |
| Limited Scalability | Hardware limitations |

---

### 2.2.12.4 Advantages

| Advantage | Explanation |
|---|---|
| Fast Communication | Shared memory speed |
| Simpler Coordination | Easier synchronization |

---

### 2.2.12.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Scalability Limits | Hardware bottlenecks |
| Contention | Shared resource conflicts |

---

## 2.2.13 Shared Disk Architecture

---

### 2.2.13.1 Definition

Shared disk architecture is:

- Multiple servers share the same storage disks but maintain separate memory.

---

### 2.2.13.2 Structure

```text
Server 1 → Shared Disk
Server 2 → Shared Disk
Server 3 → Shared Disk
```

---

### 2.2.13.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Shared Storage | Common disk system |
| Independent Memory | Each server has own RAM |
| Better Availability | Failover support |
| Coordination Required | Disk synchronization needed |

---

### 2.2.13.4 Advantages

| Advantage | Explanation |
|---|---|
| High Availability | Easier failover |
| Flexible Scaling | Additional servers possible |

---

### 2.2.13.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Disk Contention | Shared storage bottleneck |
| Complex Coordination | Cache synchronization needed |

---

## 2.2.14 Shared Nothing Architecture

---

### 2.2.14.1 Definition

Shared nothing architecture is:

- A distributed architecture where each server has its own memory and storage.

Servers operate independently.

---

### 2.2.14.2 Structure

```text
Server 1 → Own Memory → Own Disk
Server 2 → Own Memory → Own Disk
Server 3 → Own Memory → Own Disk
```

---

### 2.2.14.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Independent Nodes | Separate resources |
| High Scalability | Easy horizontal expansion |
| Fault Isolation | Failures isolated |
| Network Communication | Nodes communicate through network |

---

### 2.2.14.4 Advantages

| Advantage | Explanation |
|---|---|
| Excellent Scalability | Massive horizontal growth |
| Better Fault Isolation | Independent systems |
| High Performance | Parallel processing |

---

### 2.2.14.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Complex Data Distribution | Sharding challenges |
| Network Dependency | Communication overhead |
| Complex Transactions | Distributed transaction difficulty |

---

### 2.2.14.6 Real-World Examples

| System | Architecture |
|---|---|
| Google Bigtable | Shared nothing |
| Cassandra | Shared nothing |
| Amazon DynamoDB | Shared nothing |

---

## 2.2.15 Architecture Comparison Summary

---

### 2.2.15.1 Tier Architecture Comparison

| Architecture | Complexity | Scalability | Security | Common Usage |
|---|---|---|---|---|
| Single-tier | Low | Low | Low | Personal apps |
| Two-tier | Medium | Medium | Medium | Small business systems |
| Three-tier | High | High | High | Enterprise systems |

---

### 2.2.15.2 Distributed Architecture Comparison

| Architecture | Memory Sharing | Storage Sharing | Scalability |
|---|---|---|---|
| Shared Memory | Shared | Shared | Low |
| Shared Disk | Separate | Shared | Medium |
| Shared Nothing | Separate | Separate | Very High |

---

## 2.2.16 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Confusing architecture with database type | Architecture describes system design |
| Thinking all applications directly access databases | Modern systems use intermediate layers |
| Assuming microservices always use one database | Each service usually owns its own database |
| Ignoring scalability considerations | Architecture strongly affects scalability |
| Confusing distributed databases with cloud databases | They are related but not identical |
