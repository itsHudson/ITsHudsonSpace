# 2.6 Database Relationships

---

## 2.6.1 Introduction to Database Relationships

Database relationships define:

- How data in one table is connected to data in another table.

Relationships are one of the most important concepts in relational databases.

Without relationships:

- Data becomes isolated
- Redundancy increases
- Data consistency becomes difficult
- Complex systems become harder to manage

Database relationships help databases:

| Benefit | Explanation |
|---|---|
| Reduce Redundancy | Avoid duplicate data |
| Improve Organization | Logical data structure |
| Maintain Consistency | Connected accurate data |
| Support Scalability | Easier system growth |
| Improve Querying | Easier data retrieval |

---

## 2.6.2 Why Database Relationships are Important

Modern systems contain interconnected data.

Example:

| System | Connected Data |
|---|---|
| University System | Students ↔ Courses |
| Banking System | Customers ↔ Accounts |
| Hospital System | Patients ↔ Doctors |
| E-commerce System | Customers ↔ Orders |

Without relationships:

- Data duplication increases
- Updates become inconsistent
- Maintenance becomes difficult

---

## 2.6.3 Real-World Analogy

Think of database relationships like:

- Relationships between people in real life.

Examples:

| Real-Life Relationship | Database Equivalent |
|---|---|
| One person owns one passport | One-to-One |
| One teacher teaches many students | One-to-Many |
| Many students take many courses | Many-to-Many |

---

## 2.6.4 Core Relationship Concepts

Relationships are usually created using:

| Component | Purpose |
|---|---|
| Primary Key | Unique identifier |
| Foreign Key | Connects tables |
| Constraints | Maintains integrity |

## 2.6.6 One-to-One Relationship (1:1)

---

### 2.6.6.1 Definition

A one-to-one relationship is:

- A relationship where one record in Table A is related to only one record in Table B.

Similarly:

- One record in Table B relates to only one record in Table A.

---

### 2.6.6.2 Real-World Analogy

Think of:

- One person ↔ One passport

Each person has:

- One passport

Each passport belongs to:

- One person

---

### 2.6.6.3 Example

#### Employees Table

| EmployeeID | EmployeeName |
|---|---|
| 101 | John |

#### EmployeeDetails Table

| EmployeeID | PassportNumber |
|---|---|
| 101 | P88991 |

One employee links to one detail record.

---

### 2.6.6.4 Structure Diagram

```text
Employee ───── EmployeeDetails
    1                 1
```

---

### 2.6.6.5 Characteristics

| Characteristic | Explanation |
|---|---|
| Single Pairing | One record matches one record |
| Tight Relationship | Closely connected data |
| Often Split for Security | Sensitive data separated |

---

### 2.6.6.6 Advantages

| Advantage | Explanation |
|---|---|
| Better Organization | Separates data logically |
| Improved Security | Sensitive data isolated |
| Reduced Null Values | Optional data separated |

---

### 2.6.6.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Additional Complexity | Extra joins required |
| More Tables | Increased structure complexity |

---

### 2.6.6.8 Real-World Examples

| Example | Explanation |
|---|---|
| Person ↔ Passport | One passport per person |
| User ↔ Profile | One profile per user |
| Vehicle ↔ Registration | One registration per vehicle |

---

## 2.6.7 One-to-Many Relationship (1:M)

---

### 2.6.7.1 Definition

A one-to-many relationship is:

- A relationship where one record in Table A can relate to many records in Table B.

However:

- Each record in Table B relates to only one record in Table A.

This is the most common database relationship type.

---

### 2.6.7.2 Real-World Analogy

Think of:

- One teacher teaches many students.

But:

- Each student belongs to one teacher.

---

### 2.6.7.3 Example

#### Departments Table

| DepartmentID | DepartmentName |
|---|---|
| D01 | IT |

#### Employees Table

| EmployeeID | EmployeeName | DepartmentID |
|---|---|---|
| 101 | John | D01 |
| 102 | Sarah | D01 |

One department has many employees.

---

### 2.6.7.4 Structure Diagram

```text
Department ───── Employees
      1             Many
```

---

### 2.6.7.5 Characteristics

| Characteristic | Explanation |
|---|---|
| Most Common Relationship | Widely used |
| Parent-Child Structure | One parent, many children |
| Foreign Key Usage | Child table stores reference |

---

### 2.6.7.6 Advantages

| Advantage | Explanation |
|---|---|
| Efficient Design | Minimal redundancy |
| Logical Organization | Clear hierarchy |
| Easy Querying | Simple relationships |

---

### 2.6.7.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Parent Dependency | Child records depend on parent |
| Cascading Issues | Deletion effects possible |

---

### 2.6.7.8 Real-World Examples

| Example | Explanation |
|---|---|
| Customer ↔ Orders | One customer places many orders |
| Hospital ↔ Patients | One doctor treats many patients |
| Company ↔ Employees | One company has many employees |

---

## 2.6.8 Many-to-Many Relationship (M:N)

---

### 2.6.8.1 Definition

A many-to-many relationship is:

- A relationship where many records in Table A relate to many records in Table B.

This relationship usually requires:

- A junction table (bridge table).

---

### 2.6.8.2 Real-World Analogy

Think of:

- Many students take many courses.

Each student can take:

- Multiple courses

Each course can contain:

- Multiple students

---

### 2.6.8.3 Example

#### Students Table

| StudentID | StudentName |
|---|---|
| 101 | John |
| 102 | Sarah |

#### Courses Table

| CourseID | CourseName |
|---|---|
| C01 | Database Systems |
| C02 | Networking |

#### Enrollment Table

| StudentID | CourseID |
|---|---|
| 101 | C01 |
| 101 | C02 |
| 102 | C01 |

---

### 2.6.8.4 Structure Diagram

```text
Students ── Enrollment ── Courses
   Many           Many
```

---

### 2.6.8.5 Junction Table

The Enrollment table acts as:

| Purpose | Explanation |
|---|---|
| Relationship Connector | Links tables |
| Foreign Key Holder | Stores references |
| Relationship Manager | Controls many-to-many mapping |

---

### 2.6.8.6 Characteristics

| Characteristic | Explanation |
|---|---|
| Complex Relationship | Multiple mappings |
| Junction Table Required | Bridge table needed |
| Flexible Structure | Supports multiple connections |

---

### 2.6.8.7 Advantages

| Advantage | Explanation |
|---|---|
| Highly Flexible | Complex relationships |
| Realistic Modeling | Real-world scenarios |
| Scalable Design | Large relationship systems |

---

### 2.6.8.8 Disadvantages

| Disadvantage | Explanation |
|---|---|
| More Complex Queries | Additional joins |
| Additional Tables | More management |

---

### 2.6.8.9 Real-World Examples

| Example | Explanation |
|---|---|
| Students ↔ Courses | Multiple enrollments |
| Products ↔ Orders | Many products per order |
| Actors ↔ Movies | Multiple movie appearances |

---

## 2.6.9 Recursive Relationship

---

### 2.6.9.1 Definition

A recursive relationship is:

- A relationship where a table relates to itself.

This is also called:

- Self-referencing relationship

---

### 2.6.9.2 Real-World Analogy

Think of:

- Employees and managers.

An employee may also be another employee’s manager.

---

### 2.6.9.3 Example

#### Employees Table

| EmployeeID | EmployeeName | ManagerID |
|---|---|---|
| 101 | John | NULL |
| 102 | Sarah | 101 |
| 103 | Alex | 101 |

Employee 101 manages other employees.

---

### 2.6.9.4 Structure Diagram

```text
Employee
   ↑
Manager
```

---

### 2.6.9.5 Characteristics

| Characteristic | Explanation |
|---|---|
| Self-Referencing | Same table connected |
| Hierarchical Structure | Tree-like organization |
| Common in Organizations | Reporting structures |

---

### 2.6.9.6 Advantages

| Advantage | Explanation |
|---|---|
| Flexible Hierarchies | Organizational trees |
| Reduced Extra Tables | Self-contained structure |

---

### 2.6.9.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Complex Queries | Recursive joins |
| Difficult Maintenance | Hierarchy management |

---

### 2.6.9.8 Real-World Examples

| Example | Explanation |
|---|---|
| Employee management | Manager hierarchy |
| Folder systems | Parent-child folders |
| Product categories | Nested categories |

---

## 2.6.10 Optional Relationship

---

### 2.6.10.1 Definition

An optional relationship is:

- A relationship where a record may or may not participate in the relationship.

Participation is:

- Not required

---

### 2.6.10.2 Example

#### Employees Table

| EmployeeID | EmployeeName | ParkingSpotID |
|---|---|---|
| 101 | John | P01 |
| 102 | Sarah | NULL |

Some employees have parking spots.

Some do not.

---

### 2.6.10.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Nullable Foreign Key | Relationship may not exist |
| Flexible Participation | Optional connection |
| Common in Real Systems | Not all entities require relationships |

---

### 2.6.10.4 Advantages

| Advantage | Explanation |
|---|---|
| Greater Flexibility | Supports incomplete data |
| Easier Expansion | Optional features possible |

---

### 2.6.10.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| More Null Values | Empty references |
| Additional Validation | Business rules needed |

---

### 2.6.10.6 Real-World Examples

| Example | Explanation |
|---|---|
| Employee ↔ Parking Spot | Not everyone has parking |
| Customer ↔ Membership | Optional membership |
| User ↔ Profile Picture | Optional image |

---

## 2.6.11 Mandatory Relationship

---

### 2.6.11.1 Definition

A mandatory relationship is:

- A relationship where participation is required.

The related record must exist.

---

### 2.6.11.2 Example

#### Orders Table

| OrderID | CustomerID |
|---|---|
| O1001 | C01 |

Every order must belong to a customer.

---

### 2.6.11.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Required Participation | Relationship compulsory |
| Non-Nullable Foreign Key | Foreign key required |
| Strong Integrity | Prevents orphan records |

---

### 2.6.11.4 Advantages

| Advantage | Explanation |
|---|---|
| Strong Consistency | Better integrity |
| Prevents Invalid Data | Required relationships |

---

### 2.6.11.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Less Flexibility | Strict rules |
| More Validation Required | Data enforcement |

---

### 2.6.11.6 Real-World Examples

| Example | Explanation |
|---|---|
| Order ↔ Customer | Every order needs customer |
| Enrollment ↔ Student | Enrollment requires student |
| Invoice ↔ Payment | Payment linked to invoice |

## 2.6.14 Foreign Keys and Relationships

---

### 2.6.14.1 Definition

A foreign key is:

- A column that references the primary key of another table.

Foreign keys create relationships.

---

### 2.6.14.2 Example

#### Departments Table

| DepartmentID |
|---|
| D01 |

#### Employees Table

| EmployeeID | DepartmentID |
|---|---|
| 101 | D01 |

DepartmentID in Employees is a foreign key.

---

### 2.6.14.3 SQL Example

```sql
CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    EmployeeName VARCHAR(100),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID)
    REFERENCES Departments(DepartmentID)
);
```

## 2.6.17 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Confusing one-to-many and many-to-many | Many-to-many requires junction table |
| Forgetting foreign keys | Relationships cannot be enforced |
| Using many-to-many unnecessarily | Adds complexity |
| Ignoring optional relationships | Real systems often contain optional data |
| Not enforcing mandatory relationships | Causes inconsistent data |

- Connecting data
