# 2.12 Database Transactions Basics

---

# 2.12.1 Introduction to Database Transactions

A database transaction is:

- A sequence of one or more database operations treated as a single logical unit of work.

A transaction ensures that database operations are completed:

- Safely
- Consistently
- Reliably

Transactions are one of the most critical concepts in database systems because they protect data from:

- Corruption
- Inconsistency
- Partial updates
- System failures
- Concurrent access problems

---

# 2.12.2 Why Transactions are Important

Modern systems constantly perform transactions.

Examples:

| System | Example Transaction |
|---|---|
| Banking System | Money transfer |
| E-commerce System | Order placement |
| Hospital System | Patient registration |
| ATM System | Cash withdrawal |

Without transactions:

- Systems could store incomplete data
- Financial records could become incorrect
- Data consistency would fail

---

# 2.12.3 Real-World Analogy

Think of a transaction like:

- A complete shopping purchase process.

The process includes:

- Customer selects products
- Payment processed
- Inventory updated
- Receipt generated

If one step fails:

- The entire operation should cancel safely

Similarly, database transactions ensure:

- Either everything succeeds OR nothing changes.

---

# 2.12.4 Characteristics of Transactions

Transactions follow the famous:

- ACID properties

| Property | Meaning |
|---|---|
| Atomicity | All or nothing |
| Consistency | Valid state maintained |
| Isolation | Transactions do not interfere improperly |
| Durability | Committed data survives failures |

---

# 2.12.5 ACID Properties Overview

---

## 2.12.5.1 Atomicity

Atomicity means:

- A transaction is indivisible.

Either:

- Entire transaction succeeds

OR

- Entire transaction fails

### Example

Bank transfer:

- Deduct money from Account A
- Add money to Account B

If step 2 fails:

- Step 1 must also be reversed

---

## 2.12.5.2 Consistency

Consistency means:

- Transactions move the database from one valid state to another valid state.

Database rules remain valid.

### Example

Account balance cannot become negative if rules prohibit it.

---

## 2.12.5.3 Isolation

Isolation means:

- Multiple transactions should not interfere incorrectly with each other.

Concurrent users should not create inconsistent results.

---

## 2.12.5.4 Durability

Durability means:

- Once committed, data remains permanent even after crashes.

Committed changes survive:

- Power failures
- System crashes
- Restarts

---

# 2.12.6 Transaction Lifecycle

---

## 2.12.6.1 Definition

The transaction lifecycle describes:

- The stages a transaction goes through from start to completion.

---

## 2.12.6.2 Main Transaction States

| State | Description |
|---|---|
| Active | Transaction executing |
| Partially Committed | Final operation completed |
| Committed | Changes permanently saved |
| Failed | Error occurred |
| Aborted | Changes reversed |
| Terminated | Transaction finished |

---

## 2.12.6.3 Lifecycle Diagram

```text
Start
  ↓
Active
  ↓
Partially Committed
  ↓
Committed
  ↓
Terminated
```

If failure occurs:

```text
Active
  ↓
Failed
  ↓
Aborted
```

---

## 2.12.6.4 Step-by-Step Example

### Online Banking Transfer

### Step 1 — Transaction Starts

```sql
BEGIN TRANSACTION;
```

Transaction becomes:

- Active

### Step 2 — Money Deducted

```sql
UPDATE Accounts
SET Balance = Balance - 500
WHERE AccountID = 1001;
```

### Step 3 — Money Added

```sql
UPDATE Accounts
SET Balance = Balance + 500
WHERE AccountID = 2001;
```

### Step 4 — Commit Transaction

```sql
COMMIT;
```

Transaction becomes:

- Committed

# 2.12.7 Commit

---

## 2.12.7.1 Definition

Commit means:

- Permanently saving all transaction changes to the database.

After commit:

- Changes become permanent
- Other users can see changes
- Rollback usually becomes impossible

---

## 2.12.7.2 SQL Example

```sql
COMMIT;
```

---

## 2.12.7.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Permanent Save | Data finalized |
| Transaction Success | Operation completed |
| Durability Enforcement | Changes survive failure |

---

## 2.12.7.4 Real-World Analogy

Think of commit like:

- Clicking "Confirm Purchase" in online shopping.

Once confirmed:

- The transaction becomes official.

---

## 2.12.7.5 Example Workflow

### Step-by-Step

1. User places order
2. Inventory updated
3. Payment processed
4. COMMIT executed
5. Order permanently stored

## 2.12.7.6 Advantages

| Advantage | Explanation |
|---|---|
| Permanent Consistency | Reliable storage |
| Safe Finalization | Stable data |

## 2.12.7.7 Beginner Mistakes

|---|---|

# 2.12.8 Rollback

---

## 2.12.8.1 Definition

Rollback means:

- Undoing changes made during a transaction.

Rollback restores the database to:

- Previous consistent state

---

## 2.12.8.2 SQL Example

```sql
ROLLBACK;
```

---

## 2.12.8.3 Example Workflow

### Banking Example

### Step 1 — Deduct Money

```sql
UPDATE Accounts
SET Balance = Balance - 500
WHERE AccountID = 1001;
```

### Step 2 — System Failure Occurs

Money not added to second account.

### Step 3 — Rollback Executed

```sql
ROLLBACK;
```

Original balance restored.

---

## 2.12.8.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Undo Changes | Revert transaction |
| Failure Recovery | Restore consistency |
| Atomicity Support | All-or-nothing behavior |

---

## 2.12.8.5 Real-World Analogy

Think of rollback like:

- Canceling a form before submission.

Temporary changes disappear.

---

## 2.12.8.6 Advantages

| Advantage | Explanation |
|---|---|
| Error Recovery | Fix failed operations |
| Data Protection | Prevent inconsistent state |
| Transaction Safety | Reliable systems |

---

## 2.12.8.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Resource Usage | Rollback processing overhead |
| Long Transactions | Larger rollback cost |

# 2.12.9 Savepoints

---

## 2.12.9.1 Definition

A savepoint is:

- A marker inside a transaction that allows partial rollback.

Instead of rolling back the entire transaction:

- Database can rollback to specific point.

---

## 2.12.9.2 SQL Example

```sql
SAVEPOINT Save1;
```

---

## 2.12.9.3 Example Workflow

### Step-by-Step

### Step 1 — Start Transaction

```sql
BEGIN TRANSACTION;
```

### Step 2 — First Update

```sql
UPDATE Accounts
SET Balance = Balance - 500
WHERE AccountID = 1001;
```

### Step 3 — Create Savepoint

```sql
SAVEPOINT TransferPoint;
```

### Step 4 — Second Update Fails

### Step 5 — Rollback to Savepoint

```sql
ROLLBACK TO TransferPoint;
```

Only operations after savepoint are undone.

---

## 2.12.9.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Partial Rollback | Flexible recovery |
| Transaction Control | Intermediate recovery |
| Nested Safety | Complex workflows |

---

## 2.12.9.5 Advantages

| Advantage | Explanation |
|---|---|
| Flexible Recovery | Partial undo |
| Better Error Handling | Fine-grained control |
| Complex Transaction Support | Multi-step operations |

---

## 2.12.9.6 Real-World Analogy

Think of savepoints like:

- Save checkpoints in video games.

If failure occurs:

- You restart from checkpoint instead of beginning.

# 2.12.10 Transaction Logs

---

## 2.12.10.1 Definition

Transaction logs are:

- Files that record all database transaction changes.

Logs are critical for:

- Recovery
- Rollback
- Durability
- Crash recovery

---

## 2.12.10.2 What Transaction Logs Store

| Logged Information | Example |
|---|---|
| INSERT operations | Added rows |
| UPDATE operations | Changed values |
| DELETE operations | Removed rows |
| COMMIT actions | Successful completion |
| ROLLBACK actions | Reversed changes |

---

## 2.12.10.3 Example

Suppose query executes:

```sql
UPDATE Accounts
SET Balance = 5000
WHERE AccountID = 1001;
```

Before permanent storage:

- Change recorded in transaction log.

---

## 2.12.10.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Sequential Recording | Ordered operations |
| Recovery Support | Crash restoration |
| Durability Enforcement | Permanent protection |

---

## 2.12.10.5 Write-Ahead Logging (WAL)

Many databases use:

- Write-Ahead Logging

Meaning:

- Changes written to logs BEFORE data files updated.

This improves:

- Recovery reliability

---

## 2.12.10.6 Real-World Analogy

Think of transaction logs like:

- Flight recorder black boxes in airplanes.

Even after crashes:

- Historical actions remain available.

---

## 2.12.10.7 Advantages

| Advantage | Explanation |
|---|---|
| Crash Recovery | Restore system |
| Rollback Support | Undo operations |
| Audit Trail | Track changes |
| Durability | Protect committed data |

---

## 2.12.10.8 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Storage Usage | Large log files |
| Additional Write Overhead | Extra disk activity |

# 2.12.16 Transaction Best Practices

---

## 2.12.16.1 Keep Transactions Short

Short transactions reduce:

- Lock contention
- Resource usage

---

## 2.12.16.2 Always Handle Errors

Applications should properly:

- Rollback failed operations

---

## 2.12.16.3 Use Savepoints Carefully

Savepoints help:

- Complex workflows

but excessive savepoints increase complexity.

---

## 2.12.16.4 Monitor Transaction Logs

Large logs may indicate:

- Long-running transactions
- Poor maintenance
