# 2.16 Database Availability Basics

---

# 2.16.1 Introduction to Database Availability

Database availability refers to:

- The ability of a database system to remain operational, accessible, and usable when users or applications need it.

High availability is critical because modern systems often operate:

- 24 hours a day
- 7 days a week
- Globally across different time zones

If databases become unavailable:

- Businesses lose revenue
- Transactions fail
- Users cannot access services
- Critical operations stop

Database availability focuses on:

| Goal | Explanation |
|---|---|
| Continuous Access | Systems remain usable |
| Fault Tolerance | Failures do not stop operations |
| Recovery | Systems recover quickly |
| Redundancy | Backup resources exist |
| Reliability | Stable long-term operation |

---

# 2.16.2 Why Database Availability is Important

---

## 2.16.2.1 Real-World Impact

Imagine a banking system becoming unavailable for 2 hours.

Possible consequences:

- ATM transactions fail
- Online banking inaccessible
- Credit card processing stops
- Customer trust decreases
- Financial losses occur

For some businesses:

- Even a few minutes of downtime is extremely expensive

---

## 2.16.2.2 Real-World Analogy

Think of database availability like:

- Electricity supply to a hospital.

Hospitals require:

- Continuous power
- Backup generators
- Emergency recovery systems

Similarly, databases require:

- Backup systems
- Redundant servers
- Recovery mechanisms

# 2.16.4 Backup Systems

---

## 2.16.4.1 Definition of Backup

A backup is:

- A copy of database data stored separately for recovery purposes.

Backups help organizations restore systems after:

- Hardware failures
- Data corruption
- Cyberattacks
- Human mistakes
- Natural disasters

---

## 2.16.4.2 Purpose of Backups

Backups provide:

| Purpose | Explanation |
|---|---|
| Data Protection | Prevent permanent data loss |
| Recovery | Restore damaged systems |
| Disaster Recovery | Recover after catastrophic events |
| Business Continuity | Continue operations |

---

## 2.16.4.3 Real-World Analogy

Think of backups like:

- Photocopying important documents.

If the original document is destroyed:

- The copied version can restore information

Similarly:

- Database backups restore lost data

---

## 2.16.4.4 Types of Database Backups

| Backup Type | Explanation |
|---|---|
| Full Backup | Entire database copied |
| Incremental Backup | Only changed data copied |
| Differential Backup | Changes since last full backup |
| Snapshot Backup | Point-in-time database image |

---

# 2.16.5 Full Backup

---

## 2.16.5.1 Definition

A full backup means:

- Creating a complete copy of the entire database.

---

## 2.16.5.2 Advantages

| Advantage | Explanation |
|---|---|
| Simple Recovery | Easy restoration |
| Complete Data Copy | Entire database protected |

---

## 2.16.5.3 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Large Storage Usage | Requires more space |
| Longer Backup Time | Full copying process |

---

## 2.16.5.4 Example

```text
Database Size: 500GB
Full Backup Size: 500GB
```

---

# 2.16.6 Incremental Backup

---

## 2.16.6.1 Definition

Incremental backup means:

- Only backing up data changed since the previous backup.

---

## 2.16.6.2 Example Workflow

### Day 1

- Full backup performed.

### Day 2

- Only changed records backed up.

### Day 3

- Only new changes backed up again.

---

## 2.16.6.3 Advantages

| Advantage | Explanation |
|---|---|
| Smaller Backup Size | Less storage needed |
| Faster Backup Process | Less data copied |

---

## 2.16.6.4 Disadvantages

| Disadvantage | Explanation |
|---|---|
| More Complex Recovery | Multiple backups needed |
| Backup Chain Dependency | Missing backup affects recovery |

---

# 2.16.7 Differential Backup

---

## 2.16.7.1 Definition

Differential backup means:

- Backing up all changes since the last full backup.

---

## 2.16.7.2 Example Workflow

| Day | Backup Content |
|---|---|
| Monday | Full backup |
| Tuesday | Monday → Tuesday changes |
| Wednesday | Monday → Wednesday changes |

---

## 2.16.7.3 Advantages

| Advantage | Explanation |
|---|---|
| Faster Recovery | Fewer files needed |
| Smaller Than Full Backup | Reduced storage usage |

---

## 2.16.7.4 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Backup Size Grows Over Time | More changes accumulate |
| Larger Than Incremental | More storage needed |

---

# 2.16.8 Backup Strategy Concepts

---

## 2.16.8.1 Backup Schedule

Organizations often use:

| Schedule | Purpose |
|---|---|
| Daily Incremental | Frequent protection |

---

## 2.16.8.2 Offsite Backups

Offsite backups mean:

- Storing backup copies in different physical locations.

This protects against:

- Fire
- Flood

---

## 2.16.8.3 Cloud Backups

Modern systems often store backups in:

- AWS S3
- Azure Backup
- Google Cloud Storage

# 2.16.9 Replication

---

## 2.16.9.1 Definition of Replication

Replication means:

- Maintaining copies of the same database data across multiple servers.

Replication improves:

- Availability
- Fault tolerance
- Read performance

---

## 2.16.9.2 Basic Replication Architecture

```text
Primary Database
        ↓
Replica Database 1
Replica Database 2
```

---

## 2.16.9.3 Types of Replication

| Type | Explanation |
|---|---|
| Synchronous Replication | Immediate update synchronization |
| Asynchronous Replication | Delayed synchronization |

---

# 2.16.10 Synchronous Replication

---

## 2.16.10.1 Definition

Synchronous replication means:

- Data changes must be confirmed by replicas before transaction completion.

---

## 2.16.10.2 Advantages

| Advantage | Explanation |
|---|---|
| Strong Consistency | All replicas synchronized |
| Better Data Protection | Minimal data loss risk |

---

## 2.16.10.3 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Slower Performance | Waits for replicas |
| Network Dependency | Communication delays |

---

# 2.16.11 Asynchronous Replication

---

## 2.16.11.1 Definition

Asynchronous replication means:

- Primary server updates data first, replicas update later.

---

## 2.16.11.2 Advantages

| Advantage | Explanation |
|---|---|
| Faster Transactions | No waiting for replicas |
| Better Performance | Reduced latency |

---

## 2.16.11.3 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Possible Data Loss | Replica delay risk |
| Temporary Inconsistency | Data mismatch possible |

---

# 2.16.12 Failover

---

## 2.16.12.1 Definition of Failover

Failover means:

- Automatically switching to a backup system when the primary system fails.

Failover is critical for:

- Continuous operations
- High availability systems

---

## 2.16.12.2 Basic Failover Architecture

```text
Primary Server
      ↓ Failure
Secondary Server Takes Over
```

---

## 2.16.12.3 Real-World Analogy

Think of failover like:

- Emergency backup generators during a power outage.

When primary power fails:

- Backup power immediately activates

Similarly:

- Backup database servers take over

---

## 2.16.12.4 Types of Failover

| Type | Explanation |
|---|---|
| Automatic Failover | System switches automatically |
| Manual Failover | Administrator performs switch |

---

# 2.16.13 Automatic Failover

---

## 2.16.13.1 Definition

Automatic failover means:

- The system automatically detects failure and switches to standby servers.

---

## 2.16.13.2 Advantages

| Advantage | Explanation |
|---|---|
| Minimal Downtime | Fast recovery |
| Reduced Human Dependency | Automated response |

---

## 2.16.13.3 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Complex Configuration | More setup required |
| Risk of False Detection | Incorrect failovers possible |

---

# 2.16.14 Manual Failover

---

## 2.16.14.1 Definition

Manual failover means:

- Administrators manually activate backup systems after failures.

---

## 2.16.14.2 Advantages

| Advantage | Explanation |
|---|---|
| Better Human Control | Careful decision making |
| Lower Automation Complexity | Simpler setup |

---

## 2.16.14.3 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Longer Downtime | Human response required |
| Human Error Risk | Mistakes possible |

---

# 2.16.15 Disaster Recovery Basics

---

## 2.16.15.1 Definition of Disaster Recovery

Disaster Recovery (DR) refers to:

- Procedures and systems used to restore operations after major failures or disasters.

Disasters include:

- Data center destruction
- Cyberattacks
- Hardware failures
- Natural disasters

---

## 2.16.15.2 Goals of Disaster Recovery

| Goal | Explanation |
|---|---|
| Restore Services | Resume operations quickly |
| Minimize Data Loss | Protect business information |
| Ensure Business Continuity | Reduce operational impact |

---

## 2.16.15.3 Real-World Analogy

Think of disaster recovery like:

- Emergency evacuation and rebuilding plans for cities.

Even after major disasters:

- Recovery procedures restore operations

---

# 2.16.16 Important Disaster Recovery Metrics

---

## 2.16.16.1 Recovery Time Objective (RTO)

RTO means:

- Maximum acceptable downtime.

Example:

| System | Example RTO |
|---|---|
| Banking System | 5 minutes |
| Internal HR System | 4 hours |

---

## 2.16.16.2 Recovery Point Objective (RPO)

RPO means:

- Maximum acceptable amount of lost data.

Example:

| System | Example RPO |
|---|---|
| Stock Trading | Seconds |
| Internal Reports | Several hours |

---

# 2.16.17 Disaster Recovery Strategies

---

## 2.16.17.1 Cold Site

A cold site is:

- A backup location with minimal infrastructure.

Characteristics:

- Cheapest option
- Slowest recovery

---

## 2.16.17.2 Warm Site

A warm site is:

- A partially prepared recovery environment.

Characteristics:

- Moderate cost
- Faster recovery

---

## 2.16.17.3 Hot Site

A hot site is:

- A fully operational backup environment ready immediately.

Characteristics:

- Very expensive
- Fastest recovery

---

# 2.16.18 High Availability Basics

---

## 2.16.18.1 Definition of High Availability

High Availability (HA) means:

- Designing systems to minimize downtime and ensure continuous operation.

HA systems aim for:

- Near-continuous uptime
- Fault tolerance
- Automatic recovery

---

## 2.16.18.2 High Availability Goals

| Goal | Explanation |
|---|---|
| Minimize Downtime | Continuous operations |
| Eliminate Single Points of Failure | Redundant components |
| Fast Recovery | Immediate restoration |

---

## 2.16.18.3 Common High Availability Techniques

| Technique | Purpose |
|---|---|
| Replication | Duplicate systems |
| Failover Clusters | Backup server switching |
| Load Balancing | Traffic distribution |
| Redundant Storage | Extra data copies |
| Multiple Data Centers | Geographic protection |

---

# 2.16.19 Availability Levels

Availability is often measured as percentages.

---

## 2.16.19.1 Common Availability Targets

| Availability | Maximum Downtime Per Year |
|---|---|
| 99% | ~3.65 days |
| 99.9% | ~8.76 hours |
| 99.99% | ~52.56 minutes |
| 99.999% | ~5.26 minutes |

---

## 2.16.19.2 Meaning of “Five Nines”

99.999% availability means:

- The system is almost always operational.

This is commonly required for:

- Banking systems
- Healthcare systems
- Telecommunications
- Critical infrastructure

# 2.16.21 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Confusing backups and replication | Backups restore data, replication improves availability |
| Assuming replication replaces backups | Replicated corruption also spreads |
| Ignoring disaster recovery planning | Backups alone are insufficient |
| Thinking failover prevents all downtime | Some downtime may still occur |
| Assuming high availability means zero failures | HA reduces impact, not failures |

---

# 2.16.22 Backup vs Replication vs Failover

| Feature | Backup | Replication | Failover |
|---|---|---|---|
| Main Purpose | Data recovery | Data duplication | Service continuity |
| Real-Time | Usually no | Yes | Yes |
| Protects Against Data Loss | Yes | Partially | No |
| Improves Availability | Limited | Yes | Yes |
| Supports Disaster Recovery | Yes | Yes | Yes |
