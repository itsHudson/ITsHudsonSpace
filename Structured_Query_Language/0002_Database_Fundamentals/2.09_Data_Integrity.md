# 2.9 Data Integrity

---

## 2.9.1 Introduction to Data Integrity

Data integrity refers to:

- The accuracy, consistency, reliability, and correctness of data stored in a database.

Data integrity ensures that:

- Data remains valid
- Data relationships remain correct
- Invalid data is prevented
- Database rules are enforced

Without data integrity:

- Databases become unreliable
- Incorrect information spreads
- Reports become inaccurate
- Business operations fail
- System trust decreases

---

## 2.9.2 Why Data Integrity is Important

Modern systems depend heavily on accurate data.

Examples:

| Industry | Importance of Data Integrity |
|---|---|
| Banking | Accurate financial transactions |
| Healthcare | Correct patient information |
| E-commerce | Correct orders and payments |
| Education | Accurate student records |

Even small data errors can cause:

- Financial loss
- Legal issues
- Security risks
- Operational failures

---

## 2.9.3 Real-World Analogy

Think of data integrity like:

- Rules used in official government documents.

For example:

- Passport numbers must be unique
- Birth dates must be valid
- IDs cannot be empty
- Relationships between records must exist

Similarly, databases enforce rules to maintain valid information.

---

## 2.9.4 Main Types of Data Integrity

The major types of data integrity are:

| Integrity Type | Main Purpose |
|---|---|
| Entity Integrity | Prevent duplicate or NULL primary keys |
| Referential Integrity | Maintain valid relationships |
| Domain Integrity | Ensure valid values and data types |
| User-defined Integrity | Enforce business-specific rules |

---

## 2.9.5 Entity Integrity

---

### 2.9.5.1 Definition

Entity integrity ensures:

- Every row in a table can be uniquely identified.

This is mainly enforced using:

- Primary keys

---

### 2.9.5.2 Primary Key Enforcement

A primary key must:

| Rule | Explanation |
|---|---|
| Be unique | No duplicate values |
| Not be NULL | Every row must have identifier |

---

### 2.9.5.3 Example

#### Students Table

| StudentID | StudentName |
|---|---|
| 101 | John |
| 102 | Sarah |

StudentID uniquely identifies each student.

---

### 2.9.5.4 Invalid Example

#### Duplicate Primary Key

| StudentID | StudentName |
|---|---|
| 101 | John |
| 101 | Sarah |

This violates entity integrity.

---

### 2.9.5.5 NULL Example

| StudentID | StudentName |
|---|---|
| NULL | John |

This also violates entity integrity.

---

### 2.9.5.6 SQL Example

```sql
CREATE TABLE Students (
    StudentID INT PRIMARY KEY,
    StudentName VARCHAR(100)
);
```

The PRIMARY KEY constraint enforces:

- Uniqueness
- NOT NULL

---

### 2.9.5.7 Characteristics

| Characteristic | Explanation |
|---|---|
| Unique Identification | Every row identifiable |
| No NULL Primary Keys | Required values |
| Prevents Duplicate Records | Strong consistency |

---

### 2.9.5.8 Advantages

| Advantage | Explanation |
|---|---|
| Reliable Record Identification | Unique rows |
| Better Relationships | Supports foreign keys |
| Improved Data Consistency | Accurate records |

---

### 2.9.5.9 Real-World Examples

| Example | Explanation |
|---|---|
| Passport Number | Unique citizen ID |
| Student ID | Unique student identification |
| Order ID | Unique transaction identifier |

---

## 2.9.6 Referential Integrity

---

### 2.9.6.1 Definition

Referential integrity ensures:

- Relationships between tables remain valid and consistent.

This is enforced using:

- Foreign keys

---

### 2.9.6.2 Foreign Key Enforcement

A foreign key value must:

- Match an existing primary key value

OR

- Be NULL (if optional relationship allowed)

---

### 2.9.6.3 Example

#### Departments Table

| DepartmentID | DepartmentName |
|---|---|
| D01 | IT |

#### Employees Table

| EmployeeID | EmployeeName | DepartmentID |
|---|---|---|
| 101 | John | D01 |

DepartmentID in Employees references Departments table.

---

### 2.9.6.4 Invalid Example

| EmployeeID | EmployeeName | DepartmentID |
|---|---|---|
| 101 | John | D99 |

If D99 does not exist:

- Referential integrity is violated.

---

### 2.9.6.5 SQL Example

```sql
CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    EmployeeName VARCHAR(100),
    DepartmentID VARCHAR(10),
    FOREIGN KEY (DepartmentID)
    REFERENCES Departments(DepartmentID)
);
```

---

### 2.9.6.6 Characteristics

| Characteristic | Explanation |
|---|---|
| Table Relationships | Connected records |
| Foreign Key Validation | Valid references |
| Prevents Orphan Records | Child records require valid parents |

---

### 2.9.6.7 Cascading Operations

Referential integrity may support:

| Operation | Explanation |
|---|---|
| CASCADE DELETE | Delete related records |
| CASCADE UPDATE | Update related records |
| RESTRICT | Prevent invalid operations |

---

### 2.9.6.8 Example

#### CASCADE DELETE

If customer deleted:

- Related orders may automatically delete.

---

### 2.9.6.9 Advantages

| Advantage | Explanation |
|---|---|
| Consistent Relationships | Valid table connections |
| Prevents Invalid References | Strong integrity |
| Better Data Reliability | Accurate systems |

---

### 2.9.6.10 Real-World Examples

| Example | Explanation |
|---|---|
| Orders ↔ Customers | Orders require customers |
| Enrollment ↔ Students | Enrollment requires valid students |
| Payments ↔ Accounts | Payment linked to existing account |

## 2.9.7 Domain Integrity

---

### 2.9.7.1 Definition

Domain integrity ensures:

- Data entered into a column follows valid rules and acceptable values.

Domain integrity controls:

- Data types
- Allowed ranges
- Formats
- Constraints

---

## 2.9.8 Valid Data Types

---

### 2.9.8.1 Definition

Each column should contain:

- Appropriate data types.

---

### 2.9.8.2 Example

| Column | Valid Data Type |
|---|---|
| Age | INTEGER |
| Name | VARCHAR |
| BirthDate | DATE |

---

### 2.9.8.3 Invalid Example

Storing text inside integer column:

```text
Age = "Hello"
```

This violates domain integrity.

---

### 2.9.8.4 SQL Example

```sql
CREATE TABLE Students (
    StudentID INT,
    StudentName VARCHAR(100),
    GPA DECIMAL(3,2)
);
```

---

### 2.9.8.5 Importance

Valid data types help:

| Benefit | Explanation |
|---|---|
| Accurate Storage | Proper values |
| Efficient Queries | Better processing |
| Reduced Errors | Validation support |

---

## 2.9.9 Constraints

---

### 2.9.9.1 Definition

Constraints are:

- Rules enforced by the database to maintain valid data.

Constraints are major tools for domain integrity.

---

### 2.9.9.2 Common Constraints

| Constraint | Purpose |
|---|---|
| NOT NULL | Prevent empty values |
| UNIQUE | Prevent duplicates |
| CHECK | Validate conditions |
| DEFAULT | Provide default values |
| PRIMARY KEY | Unique row identification |
| FOREIGN KEY | Relationship validation |

---

### 2.9.9.3 NOT NULL Constraint

#### Example

```sql
CREATE TABLE Students (
    StudentName VARCHAR(100) NOT NULL
);
```

StudentName cannot be empty.

---

### 2.9.9.4 UNIQUE Constraint

#### Example

```sql
CREATE TABLE Users (
    Email VARCHAR(100) UNIQUE
);
```

Emails cannot duplicate.

---

### 2.9.9.5 CHECK Constraint

#### Example

```sql
CREATE TABLE Students (
    GPA DECIMAL(3,2),
    CHECK (GPA >= 0 AND GPA <= 4.00)
);
```

Only valid GPA values allowed.

---

### 2.9.9.6 DEFAULT Constraint

#### Example

```sql
CREATE TABLE Employees (
    Status VARCHAR(20) DEFAULT 'Active'
);
```

If no value provided:

- Active becomes default.

---

### 2.9.9.7 Characteristics

| Characteristic | Explanation |
|---|---|
| Rule Enforcement | Validation |
| Automatic Checking | DBMS validation |
| Data Protection | Prevent invalid data |

---

### 2.9.9.8 Advantages

| Advantage | Explanation |
|---|---|
| Better Data Quality | Cleaner data |
| Reduced Errors | Validation support |
| Stronger Integrity | Reliable systems |

---

### 2.9.9.9 Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Ignoring constraints | Weak validation |
| Overusing NULL values | Poor consistency |
| Using wrong data types | Invalid storage |

## 2.9.10 User-defined Integrity

---

### 2.9.10.1 Definition

User-defined integrity refers to:

- Custom business rules defined by organizations or applications.

These rules depend on:

- Business requirements
- Organizational policies
- Application logic

---

### 2.9.10.2 Examples of Business Rules

| Business Rule | Explanation |
|---|---|
| Employee age must be above 18 | HR policy |
| Order amount cannot be negative | Financial rule |
| GPA must be below 4.00 | University rule |
| Loan amount limited by salary | Banking policy |

---

### 2.9.10.3 Example SQL

```sql
CREATE TABLE Employees (
    Age INT,
    CHECK (Age >= 18)
);
```

This enforces company policy.

---

### 2.9.10.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Business-Specific | Depends on organization |
| Flexible Rules | Custom validation |
| Application-Oriented | Supports workflows |

---

### 2.9.10.5 Implementation Methods

User-defined integrity may use:

| Method | Purpose |
|---|---|
| CHECK Constraints | Simple rules |
| Triggers | Advanced automation |
| Stored Procedures | Business logic |
| Application Validation | Frontend/backend validation |

---

### 2.9.10.6 Real-World Examples

| Industry | Business Rule |
|---|---|
| Banking | Withdrawal limits |
| Hospital | Valid medicine dosage |
| University | Credit hour limits |
| E-commerce | Stock availability |

---

### 2.9.10.7 Advantages

| Advantage | Explanation |
|---|---|
| Business Accuracy | Correct workflows |
| Customized Validation | Organization-specific rules |
| Better Reliability | Prevent invalid operations |

---

### 2.9.10.8 Disadvantages

| Disadvantage | Explanation |

- A primary key

- Referential integrity

- Data quality
- Reliability

- Application layer
