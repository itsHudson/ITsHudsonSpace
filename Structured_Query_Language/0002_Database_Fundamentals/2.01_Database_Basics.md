# 2.1 Database Basics

---

## 2.1.1 Introduction to Database Basics

Databases are one of the most important components in modern computing systems.

Almost every modern application uses databases, including:

- Banking systems
- Social media platforms
- E-commerce websites
- Hospital systems
- University systems
- Mobile applications
- Enterprise business systems
- Government systems

Without databases:

- Data would be difficult to organize
- Searching information would be slow
- Data duplication would increase
- Data inconsistency would occur
- Multi-user systems would become difficult to manage

Database fundamentals help us understand:

| Area | Explanation |
|---|---|
| Data Storage | How information is stored |
| Data Management | How data is organized and maintained |
| Data Retrieval | How systems search and access data |
| Data Security | How systems protect information |
| Data Consistency | How systems maintain accurate data |
| Scalability | How systems handle growth |

---

## 2.1.2 What is a Database

---

### 2.1.2.1 Definition of a Database

A database is:

- An organized collection of related data stored electronically for easy access, management, updating, and retrieval.

A database helps systems:

- Store information
- Retrieve information
- Update information
- Delete information
- Manage relationships between data

---

### 2.1.2.2 Simple Real-World Analogy

Think of a database like:

- A digital filing cabinet

Inside the cabinet:

- Files are organized
- Information is categorized
- Documents can be searched easily
- Multiple users can access information

Similarly, a database organizes digital information systematically.

---

### 2.1.2.3 Example of Database Data

Example:

A university database may contain:

| StudentID | StudentName | Course | GPA |
|---|---|---|---|
| S101 | John | Computer Science | 3.80 |
| S102 | Sarah | Business | 3.65 |
| S103 | Alex | Engineering | 3.90 |

The database stores this information in structured form.

---

### 2.1.2.4 Characteristics of a Database

| Characteristic | Explanation |
|---|---|
| Organized | Data is structured systematically |
| Persistent | Data remains stored permanently |
| Shared | Multiple users can access data |
| Secure | Access can be controlled |
| Searchable | Data can be queried efficiently |
| Scalable | Can grow with increasing data |

---

### 2.1.2.5 Types of Data Stored in Databases

Databases can store many types of data:

| Data Type | Example |
|---|---|
| Text | Names, addresses |
| Numbers | Salary, age |
| Dates | Birthdate, order date |
| Images | Profile pictures |
| Videos | Security footage |
| Audio | Voice recordings |
| Documents | PDFs, reports |

---

### 2.1.2.6 Why Databases are Important

Without databases:

- Companies cannot manage customer records efficiently
- Banks cannot process transactions safely
- Hospitals cannot manage patient records
- Websites cannot store user accounts

Databases provide:

| Benefit | Explanation |
|---|---|
| Efficiency | Faster data processing |
| Accuracy | Reduced human error |
| Security | Controlled access |
| Reliability | Stable data storage |
| Consistency | Standardized information |
| Automation | Reduced manual work |

## 2.1.3 What is DBMS

---

### 2.1.3.1 Definition of DBMS

DBMS stands for:

- Database Management System

A DBMS is software that:

- Manages databases
- Controls access to data
- Handles data storage
- Performs queries
- Ensures security and consistency

The DBMS acts as:

- A bridge between users/applications and the database.

---

### 2.1.3.2 Simple Architecture Flow

```text
User/Application
        ↓
      DBMS
        ↓
    Database
```

The DBMS manages all communication with the database.

---

### 2.1.3.3 Responsibilities of a DBMS

| Responsibility | Explanation |
|---|---|
| Data Storage | Stores and organizes data |
| Data Retrieval | Searches and returns data |
| Security | Controls user access |
| Backup | Protects against data loss |
| Concurrency Control | Handles multiple users |
| Transaction Management | Maintains consistency |
| Integrity Enforcement | Prevents invalid data |

---

### 2.1.3.4 Real-World Analogy

Think of DBMS like:

- A librarian in a library

The librarian:

- Organizes books
- Finds requested books
- Prevents unauthorized access
- Maintains records

Similarly, a DBMS manages data inside databases.

---

### 2.1.3.5 Examples of DBMS Software

| DBMS | Type |
|---|---|
| MySQL | Relational DBMS |
| PostgreSQL | Relational DBMS |
| Oracle Database | Enterprise DBMS |
| Microsoft SQL Server | Relational DBMS |
| MongoDB | NoSQL DBMS |
| SQLite | Embedded DBMS |

---

### 2.1.3.6 Functions of a DBMS

#### Step-by-Step Example

Suppose a user wants to find student data.

#### Step 1 — User Sends Query

```sql
SELECT * FROM Students;
```

#### Step 2 — DBMS Processes Query

The DBMS:

- Checks syntax
- Checks permissions
- Searches storage

#### Step 3 — Database Returns Data

| StudentID | StudentName |
|---|---|
| S101 | John |
| S102 | Sarah |

#### Step 4 — DBMS Sends Result to User

The user receives requested information.

## 2.1.4 What is RDBMS

---

### 2.1.4.1 Definition of RDBMS

RDBMS stands for:

- Relational Database Management System

An RDBMS stores data using:

- Tables
- Rows
- Columns
- Relationships

The word "Relational" means:

- Data tables are related to one another.

---

### 2.1.4.2 Example of Relational Structure

#### Students Table

| StudentID | StudentName |
|---|---|
| 101 | John |
| 102 | Sarah |

#### Courses Table

| CourseID | CourseName |
|---|---|
| C01 | Database Systems |
| C02 | Networking |

#### Enrollment Table

| StudentID | CourseID |
|---|---|
| 101 | C01 |
| 102 | C02 |

Relationships connect the tables together.

---

### 2.1.4.3 Characteristics of RDBMS

| Characteristic | Explanation |
|---|---|
| Table-based | Data stored in tables |
| Structured | Fixed schema |
| Relational | Tables linked using keys |
| SQL Support | Uses SQL language |
| ACID Support | Reliable transactions |
| Integrity Enforcement | Maintains data accuracy |

---

### 2.1.4.4 Real-World Analogy

Think of RDBMS like:

- Multiple Excel sheets connected together.

Each sheet contains related information.

Relationships allow systems to connect records across sheets.

---

### 2.1.4.5 Examples of RDBMS

| RDBMS | Vendor |
|---|---|
| MySQL | Oracle |
| PostgreSQL | PostgreSQL Global Development Group |
| Oracle Database | Oracle |
| SQL Server | Microsoft |
| MariaDB | MariaDB Foundation |

---

### 2.1.4.6 DBMS vs RDBMS

| Feature | DBMS | RDBMS |
|---|---|---|
| Full Form | Database Management System | Relational Database Management System |
| Storage Method | Various methods | Tables |
| Relationships | May not support | Strong relationship support |
| Normalization | Limited | Strong support |
| SQL Usage | Optional | Standard |
| Multi-user Support | Basic | Advanced |
| Integrity Constraints | Limited | Strong |

---

## 2.1.5 Distributed Database

---

### 2.1.5.1 Definition

A distributed database is:

- A database where data is stored across multiple physical locations or servers.

The servers may exist:

- In different cities
- In different countries
- Across cloud environments

---

### 2.1.5.2 Real-World Analogy

Think of distributed databases like:

- Multiple bank branches sharing customer information.

Even though data exists in different locations:

- All branches work together
- Customers can access services anywhere

---

### 2.1.5.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Multiple Servers | Data spread across systems |
| Network Communication | Servers communicate through networks |
| Scalability | Easier to expand |
| Fault Tolerance | One server failure may not stop system |
| Parallel Processing | Faster large-scale processing |

---

### 2.1.5.4 Advantages

| Advantage | Explanation |
|---|---|
| High Availability | System continues operating during failures |
| Scalability | Easier horizontal growth |
| Faster Local Access | Nearby users get faster access |
| Load Distribution | Workload spread across servers |

---

### 2.1.5.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Complexity | Harder to manage |
| Synchronization Issues | Data consistency challenges |
| Network Dependency | Requires reliable networking |
| Higher Cost | More infrastructure needed |

---

### 2.1.5.6 Real-World Examples

| Company | Usage |
|---|---|
| Google | Global distributed systems |
| Amazon | Worldwide e-commerce infrastructure |
| Facebook | Global social media data |
| Netflix | Distributed content systems |

## 2.1.6 Centralized Database

---

### 2.1.6.1 Definition

A centralized database is:

- A database stored and managed in a single central location.

All users connect to:

- One server
- One central database system

---

### 2.1.6.2 Real-World Analogy

Think of:

- A school office storing all student records in one room.

Everyone must access the same central location.

---

### 2.1.6.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Single Server | Centralized management |
| Easier Control | Simpler administration |
| Consistent Data | One source of truth |
| Lower Complexity | Easier maintenance |

---

### 2.1.6.4 Advantages

| Advantage | Explanation |
|---|---|
| Easier Security | Single control point |
| Easier Backup | Centralized backups |
| Easier Maintenance | Simpler updates |
| Lower Complexity | Easier system management |

---

### 2.1.6.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Single Point of Failure | Server failure affects all users |
| Scalability Limitations | Harder to scale |
| Network Bottleneck | Heavy traffic may slow system |

## 2.1.8 Embedded Database

---

### 2.1.8.1 Definition

An embedded database is:

- A lightweight database integrated directly into an application.

It usually:

- Runs locally
- Requires minimal configuration
- Does not need a separate database server

---

---

- A database designed for day-to-day business operations and transactions.

Also called:

- OLTP database
- Transactional database

---

### 2.1.9.2 Examples of Operational Systems

| System | Purpose |
|---|---|
| Banking System | Financial transactions |
| ATM System | Cash withdrawals |
| E-commerce Website | Customer orders |
| Hospital System | Patient records |

---

### 2.1.9.3 Characteristics

| Characteristic | Explanation |
|---|---|
| High Concurrency | Many users simultaneously |
| Fast Transactions | Quick insert/update |
| Real-Time Processing | Immediate operations |
| Strong Consistency | Accurate transactions |

---

### 2.1.9.4 Example Workflow

#### Online Shopping System

## 2.1.10 Analytical Database

---

### 2.1.10.1 Definition

An analytical database is:

- A database optimized for analysis, reporting, and large-scale data queries.

Also called:

- OLAP database
- Data warehouse system

---

### 2.1.10.2 Purpose of Analytical Databases

Analytical databases help organizations:

- Analyze trends
- Generate reports
- Perform business intelligence
- Support decision making

---

### 2.1.10.3 Examples

| Industry | Analysis Example |
|---|---|
| Retail | Sales trend analysis |
| Banking | Fraud analysis |
| Healthcare | Patient analytics |
| Marketing | Customer behavior analysis |

---

### 2.1.10.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Large Data Scans | Processes huge datasets |
| Read-Heavy | Mostly querying |
| Historical Data | Stores long-term information |
| Complex Queries | Aggregations and analytics |

---

### 2.1.10.5 Operational Database vs Analytical Database

| Feature | Operational Database | Analytical Database |
|---|---|---|
| Main Purpose | Daily transactions | Data analysis |
| Workload Type | OLTP | OLAP |
| Query Type | Simple fast queries | Complex analytical queries |
| Data Volume | Current operational data | Historical large-scale data |
| Updates | Frequent | Less frequent |
| Users | End users/applications | Analysts/managers |

## 2.1.12 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Confusing DBMS and Database | Database stores data, DBMS manages it |
| Thinking RDBMS and DBMS are identical | RDBMS is a specialized DBMS |
| Assuming all databases use tables | NoSQL databases may not |
| Confusing OLTP and OLAP | Transaction vs analytics systems |
| Ignoring scalability differences | Different databases solve different problems |
