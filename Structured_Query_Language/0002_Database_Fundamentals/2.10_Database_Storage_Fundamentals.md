# 2.10 Database Storage Fundamentals

---

## 2.10.1 Introduction to Database Storage Fundamentals

Database storage fundamentals explain:

- How databases physically store, organize, manage, and retrieve data inside storage systems.

Although users usually see:

- Tables
- Rows
- Columns

Internally, databases store data using:

- Files
- Pages
- Blocks
- Segments
- Memory structures

Understanding storage fundamentals is extremely important because storage design directly affects:

| Area | Impact |
|---|---|
| Query Performance | Faster or slower queries |
| Scalability | Ability to handle growth |
| Storage Efficiency | Disk usage optimization |
| Recovery | Backup and restoration |
| Transaction Processing | Logging and consistency |
| Analytics Performance | Reporting speed |

---

## 2.10.2 Why Database Storage Fundamentals are Important

Modern databases manage:

- Millions of records
- Terabytes of data
- High transaction workloads
- Real-time systems

Without proper storage management:

- Queries become slow
- Storage becomes inefficient
- Recovery becomes difficult
- Systems become unstable

Understanding storage helps with:

| Purpose | Explanation |
|---|---|
| Performance Tuning | Faster queries |
| Capacity Planning | Storage forecasting |
| Troubleshooting | Diagnose bottlenecks |
| Database Administration | Better maintenance |
| System Design | Proper architecture |

---

## 2.10.3 Real-World Analogy

Think of database storage like:

- A large warehouse system.

| Warehouse Concept | Database Equivalent |
|---|---|
| Warehouse building | Database files |
| Shelves | Segments |
| Boxes | Extents |
| Compartments | Pages/Blocks |
| Individual products | Rows |

The database organizes data in layers for efficiency.

---

## 2.10.4 Physical Storage

Physical storage refers to:

- The actual files and structures used to store database information on storage devices.

Database systems commonly use:

| Storage Component | Purpose |
|---|---|
| Data Files | Store actual data |
| Log Files | Store transaction changes |
| Temp Files | Store temporary operations |

---

## 2.10.5 Data Files

---

### 2.10.5.1 Definition

Data files are:

- Physical files that store the actual database data.

These files contain:

- Tables
- Rows
- Indexes
- Metadata

---

### 2.10.5.2 Characteristics

| Characteristic | Explanation |
|---|---|
| Persistent Storage | Data survives reboot |
| Managed by DBMS | Database controls files |
| Structured Organization | Organized internally |
| Large Capacity | Stores massive datasets |

---

### 2.10.5.3 Example

Suppose a Students table contains:

| StudentID | StudentName |
|---|---|
| 101 | John |

This data is physically stored inside:

- Data files

---

### 2.10.5.4 Common File Types

| DBMS | Example Data File |
|---|---|
| SQL Server | .mdf |
| Oracle | .dbf |
| MySQL InnoDB | .ibd |
| SQLite | .sqlite |

---

### 2.10.5.5 Advantages

| Advantage | Explanation |
|---|---|
| Persistent Storage | Long-term data |
| Structured Access | Efficient retrieval |
| Large Data Capacity | Massive storage support |

---

### 2.10.5.6 Real-World Analogy

Think of data files like:

- Filing cabinets storing official documents.

The cabinet contains actual information used daily.

---

## 2.10.6 Log Files

---

### 2.10.6.1 Definition

Log files are:

- Files that record database transactions and changes.

Log files track operations such as:

- INSERT
- UPDATE
- DELETE
- COMMIT
- ROLLBACK

---

### 2.10.6.2 Purpose of Log Files

Log files help databases:

| Purpose | Explanation |
|---|---|
| Recovery | Restore after crash |
| Rollback | Undo failed transactions |
| Consistency | Maintain ACID properties |
| Auditing | Track modifications |

---

### 2.10.6.3 Example

Suppose user updates account balance:

```sql
UPDATE Accounts
SET Balance = 5000
WHERE AccountID = 1001;
```

The operation is first recorded in:

- Log files

---

### 2.10.6.4 Characteristics

| Characteristic | Explanation |
|---|---|
| Sequential Recording | Changes logged in order |
| Critical for Recovery | Supports restoration |
| Write-Intensive | Constant updates |

---

### 2.10.6.5 Real-World Analogy

Think of log files like:

- Security camera recordings.

Even if something fails:

- The history remains available.

---

### 2.10.6.6 Types of Logs

| Log Type | Purpose |
|---|---|
| Transaction Log | Track transactions |
| Redo Log | Reapply changes |
| Undo Log | Reverse changes |
| Audit Log | Security tracking |

---

## 2.10.7 Temp Files

---

### 2.10.7.1 Definition

Temp files are:

- Temporary storage files used during database operations.

They help with:

- Sorting
- Joins
- Temporary query processing
- Intermediate calculations

---

### 2.10.7.2 Example

Suppose a large query performs:

```sql
ORDER BY StudentName
```

The DBMS may temporarily store sorted data in:

- Temp files

---

### 2.10.7.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Temporary Usage | Short-term storage |
| Query Processing Support | Intermediate operations |
| Automatically Managed | Controlled by DBMS |

---

### 2.10.7.4 Advantages

| Advantage | Explanation |
|---|---|
| Improves Query Processing | Handles large operations |
| Supports Complex Queries | Temporary calculations |

---

### 2.10.7.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Additional Disk Usage | Temporary storage overhead |
| Performance Bottlenecks | Large temp usage slows system |

---

### 2.10.7.6 Real-World Analogy

Think of temp files like:

- Temporary working desks used while organizing documents.

After work completes:

- Temporary materials are removed.

---

## 2.10.8 Storage Units

Storage units are:

- Internal structures used by databases to organize data efficiently.

The major storage units are:

| Storage Unit | Purpose |
|---|---|
| Pages | Smallest storage unit |
| Blocks | Database I/O unit |
| Extents | Group of pages/blocks |
| Segments | Large storage allocation |

---

## 2.10.9 Pages

---

### 2.10.9.1 Definition

A page is:

- The smallest unit of data storage and retrieval in many databases.

Databases read and write data:

- By pages

Not:

- One row at a time

---

### 2.10.9.2 Example

Suppose one page size is:

- 8 KB

That page may contain:

- Multiple rows

---

### 2.10.9.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Small Storage Unit | Basic storage structure |
| Fixed Size | Predefined size |
| I/O Optimization | Efficient reading/writing |

---

### 2.10.9.4 Real-World Analogy

Think of pages like:

- Pages inside a book.

Each page contains:

- Multiple pieces of information

---

### 2.10.9.5 Common Page Sizes

| DBMS | Common Page Size |
|---|---|
| SQL Server | 8 KB |
| PostgreSQL | 8 KB |
| Oracle | 2 KB – 32 KB |

---

## 2.10.10 Blocks

---

### 2.10.10.1 Definition

A block is:

- A unit of storage used for physical database I/O operations.

Some DBMS use:

- Page and block interchangeably

Others distinguish them.

---

### 2.10.10.2 Characteristics

| Characteristic | Explanation |
|---|---|
| Physical I/O Unit | Disk reading/writing |
| Fixed Size | Defined by DBMS |
| Efficient Data Access | Bulk operations |

---

### 2.10.10.3 Real-World Analogy

Think of blocks like:

- Storage boxes containing multiple documents.

---

### 2.10.10.4 Page vs Block

| Feature | Page | Block |
|---|---|---|
| Usage | Logical storage | Physical I/O |
| DBMS Dependency | Varies | Varies |
| Granularity | Small unit | Disk operation unit |

---

## 2.10.11 Extents

---

### 2.10.11.1 Definition

An extent is:

- A collection of contiguous pages or blocks allocated together.

Extents improve:

- Storage efficiency
- Allocation performance

---

### 2.10.11.2 Structure Example

```text
Extent
 ├── Page 1
 ├── Page 2
 ├── Page 3
 └── Page 4
```

---

### 2.10.11.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Group Allocation | Multiple pages together |
| Reduces Fragmentation | Better storage management |
| Improves Performance | Efficient allocation |

---

### 2.10.11.4 Real-World Analogy

Think of extents like:

- Bundles of folders allocated together in a warehouse.

---

## 2.10.12 Segments

---

### 2.10.12.1 Definition

A segment is:

- A larger storage structure consisting of multiple extents.

Segments store:

- Tables
- Indexes
- Large objects

---

### 2.10.12.2 Structure Example

```text
Segment
 ├── Extent 1
 ├── Extent 2
 ├── Extent 3
```

---

### 2.10.12.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Large Storage Unit | Major allocation structure |
| Contains Extents | Hierarchical organization |
| Object-Oriented | Stores DB objects |

---

### 2.10.12.4 Real-World Analogy

Think of segments like:

- Entire warehouse sections containing groups of storage shelves.

---

## 2.10.13 Storage Hierarchy

---

### 2.10.13.1 Storage Hierarchy Structure

```text
Database
   ↓
Segments
   ↓
Extents
   ↓
Pages / Blocks
   ↓

```

## 2.10.14 Row-Oriented Storage

---

### 2.10.14.1 Definition

Row-oriented storage stores:

- All column values of a row together physically.

This is the traditional storage format used in most transactional databases.

---

### 2.10.14.2 Example

#### Table

| ID | Name | GPA |
|---|---|---|
| 101 | John | 3.80 |

Stored physically as:

```text
[101 | John | 3.80]
```

Entire row stored together.

---

### 2.10.14.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Row-Based Storage | Entire rows together |
| Fast Row Retrieval | Efficient transactions |
| OLTP Optimized | Transaction systems |

---

### 2.10.14.4 Advantages

| Advantage | Explanation |
|---|---|
| Fast Inserts | Efficient transactional writes |
| Fast Row Access | Retrieve complete records quickly |
| Better OLTP Performance | Transaction optimized |

---

### 2.10.14.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Slower Analytics | Large scans inefficient |
| More Disk I/O for Partial Queries | Reads unnecessary columns |

---

### 2.10.14.6 Real-World Usage

| System | Usage |
|---|---|
| Banking systems | Transaction processing |
| E-commerce systems | Order management |
| Hospital systems | Patient transactions |

---

## 2.10.15 Column-Oriented Storage

---

### 2.10.15.1 Definition

Column-oriented storage stores:

- Values from the same column together physically.

Instead of storing full rows together.

---

### 2.10.15.2 Example

#### Table

| ID | Name | GPA |
|---|---|---|
| 101 | John | 3.80 |
| 102 | Sarah | 3.90 |

Stored physically as:

#### ID Column

```text
101, 102
```

#### Name Column

```text
John, Sarah
```

#### GPA Column

```text
3.80, 3.90
```

---

### 2.10.15.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Column-Based Storage | Same-column grouping |
| Analytical Optimization | Fast aggregations |
| Compression Friendly | Similar values compress well |

---

### 2.10.15.4 Advantages

| Advantage | Explanation |
|---|---|
| Faster Analytics | Efficient large scans |
| Better Compression | Similar data grouped |
| Reduced Disk I/O | Reads only required columns |

---

### 2.10.15.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Slower Row Inserts | Transaction overhead |
| Less Efficient OLTP | Full row reconstruction needed |

---

### 2.10.15.6 Real-World Usage

| System | Usage |
|---|---|
| Data warehouses | Analytics |
| Business intelligence systems | Reporting |
| Big data analytics | Large-scale analysis |

---

## 2.10.16 Row Storage vs Column Storage

| Feature | Row-Oriented | Column-Oriented |
|---|---|---|
| Storage Style | Entire rows together | Columns together |
| Best For | OLTP | OLAP |
| Insert Speed | Faster | Slower |
| Analytical Queries | Slower | Faster |
| Compression | Lower | Higher |
| Transaction Systems | Excellent | Less suitable |

---

## 2.10.17 Real-World Full Example

---

### 2.10.17.1 Banking Transaction System

#### Row-Oriented Storage

Banking systems use row storage because transactions require:

- Fast inserts
- Fast updates
- Full record retrieval

---

### 2.10.17.2 Data Warehouse System

#### Analytics Systems

Analytics systems use column storage because reports require:

- Large scans
- Aggregations
- Fast analytics

---

### 2.10.17.3 Storage Flow Example

```text
Table Data
   ↓
Stored in Pages
   ↓
Pages grouped into Extents
   ↓
Extents grouped into Segments
   ↓
Segments stored in Data Files
```

## 2.10.18 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Thinking tables are stored exactly as displayed | Databases use internal storage structures |
| Confusing pages and rows | Pages contain multiple rows |
| Assuming row storage is always better | Depends on workload |
| Ignoring log files | Critical for recovery |
| Believing temp files are unimportant | Large queries heavily depend on them |

---

### 2.10.19.1 Choose Correct Storage Model

| Workload | Recommended Storage |
|---|---|
| OLTP | Row-oriented |
| OLAP | Column-oriented |

---

### 2.10.19.2 Monitor Log File Growth

Large logs can:

- Consume storage
- Reduce performance

---

### 2.10.19.3 Optimize Temporary Storage

Large sorting operations require:

- Efficient temp file configuration

---

### 2.10.19.4 Use Partitioning for Large Tables

Partitioning improves:

- Scalability
- Maintenance
- Query performance
