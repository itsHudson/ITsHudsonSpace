# 2.3 Database Models

---

## 2.3.1 Introduction to Database Models

A database model defines:

- The logical structure and organization of data inside a database system.

Database models determine:

- How data is stored
- How data is related
- How users access data
- How systems process relationships
- How queries are performed

Different database models are designed for different purposes.

Some models are better for:

- Structured business systems
- Complex relationships
- Large-scale analytics
- Flexible document storage
- Real-time applications

---

## 2.3.2 Why Database Models are Important

Choosing the correct database model affects:

| Area | Impact |
|---|---|
| Performance | Query speed and scalability |
| Flexibility | Ability to adapt to changes |
| Complexity | Ease of development |
| Storage Efficiency | How efficiently data is stored |
| Scalability | Ability to grow |
| Data Relationships | How connections are managed |

Poor model selection can cause:

- Slow performance
- Difficult maintenance
- Complex queries
- Scalability problems
- High operational cost

---

## 2.3.3 Real-World Analogy

Think of database models like:

- Different methods of organizing a library.

Some libraries organize books:

- By category
- By author
- By relationships
- By flexible tagging systems

Similarly, database models organize data differently depending on system requirements.

---

## 2.3.4 Classification of Database Models

Database models are generally divided into:

| Category | Description |
|---|---|
| Traditional Models | Early foundational database structures |

---

## 2.3.5 Traditional Database Models

Traditional database models were created before modern internet-scale systems.

They formed the foundation of modern databases.

The major traditional models are:

| Model | Main Structure |
|---|---|
| Relational Model | Tables |
| Hierarchical Model | Tree structure |
| Network Model | Graph-like connections |

---

## 2.3.6 Relational Model

---

### 2.3.6.1 Definition

The relational model is:

- A database model that stores data in tables consisting of rows and columns.

This is the most widely used database model today.

The relational model was proposed by:

- Edgar F. Codd (1970)

---

### 2.3.6.2 Core Structure

The relational model uses:

| Component | Description |
|---|---|
| Table | Collection of related data |
| Row | Single record |
| Column | Data attribute |
| Primary Key | Unique identifier |
| Foreign Key | Relationship connector |

---

### 2.3.6.3 Example

#### Students Table

| StudentID | StudentName | Age |
|---|---|---|
| 101 | John | 21 |
| 102 | Sarah | 22 |

#### Courses Table

| CourseID | CourseName |
|---|---|
| C01 | Database Systems |
| C02 | Networking |

---

### 2.3.6.4 Relationships

Tables are connected using keys.

Example:

| StudentID | CourseID |
|---|---|
| 101 | C01 |
| 102 | C02 |

This creates relationships between tables.

---

### 2.3.6.5 Characteristics

| Characteristic | Explanation |
|---|---|
| Table-Based | Structured rows and columns |
| Schema-Oriented | Fixed structure |
| SQL Support | Uses SQL language |
| Strong Consistency | ACID compliance |
| Relationship Support | Tables linked together |

---

### 2.3.6.6 Advantages

| Advantage | Explanation |
|---|---|
| Easy Understanding | Simple tabular structure |
| Strong Data Integrity | Constraints and relationships |
| Powerful Querying | SQL support |
| Mature Technology | Widely supported |

---

### 2.3.6.7 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Rigid Schema | Structure changes may be difficult |
| Scaling Challenges | Horizontal scaling harder |
| Complex Joins | Large joins can slow performance |

---

### 2.3.6.8 Real-World Usage

| Industry | Usage |
|---|---|
| Banking | Transaction systems |
| Hospitals | Patient management |
| Universities | Student systems |
| E-commerce | Inventory and orders |

---

### 2.3.6.9 Popular Relational Databases

| Database | Type |
|---|---|
| MySQL | RDBMS |
| PostgreSQL | RDBMS |
| Oracle Database | Enterprise RDBMS |
| SQL Server | Enterprise RDBMS |

## 2.3.7 Hierarchical Model

---

### 2.3.7.1 Definition

The hierarchical model is:

- A database model that organizes data in a tree-like parent-child structure.

Each child record has:

- One parent
- Multiple children possible

---

### 2.3.7.2 Structure

```text
Company
 ├── Department A
 │      ├── Employee 1
 │      └── Employee 2
 └── Department B
        ├── Employee 3
        └── Employee 4
```

---

### 2.3.7.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Tree Structure | Parent-child hierarchy |
| One-to-Many Relationships | One parent to many children |
| Fast Traversal | Efficient hierarchical access |
| Rigid Structure | Limited flexibility |

---

### 2.3.7.4 Advantages

| Advantage | Explanation |
|---|---|
| Simple Hierarchical Data | Easy for tree structures |
| Fast Parent-Child Access | Efficient navigation |
| High Performance | Simple relationship traversal |

---

### 2.3.7.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Inflexible Structure | Hard to modify |
| Complex Many-to-Many Relationships | Poor support |
| Data Redundancy | Possible duplication |

---

### 2.3.7.6 Real-World Examples

| Example | Explanation |
|---|---|
| Organizational charts | Company structure |
| File systems | Folder hierarchy |
| XML documents | Nested structure |

---

## 2.3.8 Network Model

---

### 2.3.8.1 Definition

The network model is:

- A database model where records are connected using multiple relationships.

Unlike hierarchical models:

- A child can have multiple parents.

---

### 2.3.8.2 Structure

```text
Student
   ↕
Enrollment
   ↕
Course
```

---

### 2.3.8.3 Characteristics

| Characteristic | Explanation |
|---|---|
| Graph-Like Structure | Multiple relationships |
| Many-to-Many Support | Strong relationship flexibility |
| Pointer-Based Navigation | Direct connections |
| Complex Structure | Harder management |

---

### 2.3.8.4 Advantages

| Advantage | Explanation |
|---|---|
| Flexible Relationships | Supports complex connections |
| Efficient Navigation | Direct record linking |
| Better Many-to-Many Support | Strong relationship modeling |

---

### 2.3.8.5 Disadvantages

| Disadvantage | Explanation |
|---|---|
| Complex Design | Hard to manage |
| Difficult Maintenance | Relationship complexity |
| Less User Friendly | Harder for beginners |

---

### 2.3.8.6 Real-World Examples

| Example | Explanation |
|---|---|
| Transportation networks | Route systems |
| Telecommunications | Network connections |
| Supply chains | Complex relationships |

- A database model that stores data as objects similar to object-oriented programming.

- Methods
- Relationships

## 2.3.17 SQL vs NoSQL Models

---

### 2.3.17.1 SQL Models

Usually include:

- Relational databases
- Structured schemas
- ACID transactions

Examples:

- MySQL
- PostgreSQL
- Oracle

---

### 2.3.17.2 NoSQL Models

Usually include:

- Document databases
- Key-value databases
- Graph databases
- Time-series systems

Examples:

- MongoDB
- Redis
- Cassandra
- Neo4j

---

### 2.3.17.3 SQL vs NoSQL Comparison

| Feature | SQL | NoSQL |
|---|---|---|
| Schema | Fixed | Flexible |
| Scaling | Vertical | Horizontal |
| Relationships | Strong | Varies |
| Transactions | Strong ACID | Often eventual consistency |
| Flexibility | Lower | Higher |

---

## 2.3.18 Common Beginner Mistakes

| Mistake | Explanation |
|---|---|
| Thinking all databases use tables | Many NoSQL databases do not |
| Assuming NoSQL means “no relationships” | Some NoSQL systems support relationships |
| Confusing database model with database software | Model is the structure concept |
| Thinking relational databases are outdated | They are still dominant |
| Assuming one model fits all systems | Different systems require different models |
